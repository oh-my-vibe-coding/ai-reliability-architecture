# Unit 4 · Week 3 · 合成评审与失效放大分析

> [← Unit 4 总览](总览.md)  ·  [← 返回目录](../../README.md)

## 本周目标

合成 **Unit 4 最终产出：对 Agent 系统的复合可靠性评审文档**，重点加上**失效放大分析**——一个小错怎么变成大错。

## 任务清单

### 阅读 · B3 · 45 分钟（无 AI）

**不读新材料**。回读：

- 你自己 W1 的 step 拆解 + 正确率估算
- 你自己 W2 的 verifier 设计
- Chip Huyen《Agents》关于 error amplification 的段落

**找**：W1 的估算现在看是否太乐观或悲观？

### 产出 · B2 · 90-120 分钟

### 合成文档结构

**标题**：`<系统名> · 复合可靠性评审 v1`

#### 1 · 系统概述（100 字）
- Agent 做什么
- 规模 / 阶段（prototype / MVP / 生产）

#### 2 · Step 拆解图（来自 W1）

#### 3 · 每 Step 正确率估算（来自 W1，可能修正）
- 表格 + 估值依据
- 标注哪些是**有数据支撑**，哪些是**估计**

#### 4 · 端到端正确率（两个数字）

- **Baseline**（无 verifier）
- **优化后**（有 W2 设计的 verifier）

#### 5 · 失效放大分析（**本周核心**）

找出至少 **2 个级联场景**，描述错误怎么从小变大。

**场景模板**：
```
场景 X · [名字]
───────────────
起点: Step 2 有 20% 概率把文档检索错
放大路径:
  - Step 3 基于错文档生成错误方案
  - Step 4 用错误方案去 tool 验证
  - Tool 返回"验证通过"（因为语法对但语义错）
  - Step 5 没做额外检查
  - 输出给用户看起来合理的错答案
最终影响: 用户执行错误方案 → 生产事故
根因: Step 2 错 + Step 4 verifier 只校语法不校语义
缓解: (a) Step 2 加 retrieval 校验; (b) Step 4 改用语义 verifier
```

#### 6 · Verifier / Gate 方案（来自 W2）
- 每个 verifier 的位置和预期收益
- 成本 / 延迟账

#### 7 · Step 预算

**业务约束**：一次 Agent 调用最多能走几步？
- 延迟预算（如 p99 ≤ 30s → 最多 N step）
- 成本预算（如 ≤ $1/req → 最多 M step）
- Token 预算（见 [深入 04](../../深入/04-为什么简单你好也消耗数万token.md)）

给出 Step 数上限，超过要**截断 / 降级**。

#### 8 · Rollback 和降级策略

Agent 跑飞了怎么办：
- 硬上限触发（step / 时间 / 成本）
- 用户能否中止
- 失败时降级到简单 baseline（如 FAQ / 人工）

#### 9 · 遗留风险 + 红队发现

至少 3 条**这套方案扛不住**的场景：
- 输入异常长
- Tool 所依赖的外部 API 慢 / 挂
- 模型升级导致某些 p_i 变化

### AI 挑错 + 红队（关键）

**挑错**：
> "数字估算里最薄弱的是哪一步？这份评审最大的盲点？"

**红队**：
> "假装你是个想让这个 Agent 翻车的攻击者。基于我的 step 图和 verifier，设计 3 种攻击方式。"

根据红队结果补充防御。

### 预测 · B1 · 每日 5 分钟

本周每次听到"某 Agent 表现不好"，猜：
- "哪一步最可能是元凶？"
- "有失效放大吗？"
