# Unit 2 · Trace-Eval 统一可观测性（4 周）

> [← 返回目录](../../README.md)  ·  对应知识章节：[第 8 章](../../知识/07-质量可观测性与DataFlywheel.md)

## 目标
理解 **trace = eval dataset = 生产监控**是一套系统，设计三层 eval，给 eval 自己设 SLO。

## 骨架 / 载体

| 骨架（训练意图，永久） | 载体（当期工具，可替换） |
|---|---|
| "trace = eval dataset = 生产监控"是一套系统的认知 | 具体可观测平台（Langfuse / LangSmith / 自建）|
| 三层 eval 设计；给 eval 自己设 SLO 与 fallback | 当期 Judge 模型、标注工具 |
| Judge 校准方法（Cohen's κ / agreement rate）、Data Flywheel 的 owner 链 | 具体 trace SDK、数据仓 / 标注平台 |

判断标准看左列：三层 eval 与 Flywheel 的结构不随工具变；右列的平台、Judge 模型会月度漂移，选型对比时以当期实测为准。

## 产出
统一观测 + 评估管线**设计图 + 工具选型 PoC 方案**，包含：
- 三层 eval 体系的具体指标
- eval pipeline 与 trace pipeline 的合一架构
- 选型对比（Langfuse / LangSmith / 自建）
- Data Flywheel 的运作流程图

**Bloom 目标层级**：**综合（Create）** — 整合 trace / eval / 数据回流的系统设计。

## Mastery Gate

达到以下标准才算 Unit 2 mastered：

- [ ] 三层 eval 每层都有 **≥3 条具体可执行检查项**（不是抽象描述）
- [ ] 能画出 **"trace → eval dataset"** 的统一数据流图，说明为什么不能分两套
- [ ] Judge 模型选型 + **校准方法**说清楚（Cohen's κ 或 agreement rate）
- [ ] 选型对比**至少 3 个工具**，每个给出**劣势**（不是只讲好处）
- [ ] Data Flywheel 循环**每一步都有 owner**（谁负责数据回流、标注、变更）
- [ ] Eval 自身 SLO 定义好，**Eval 挂了的 fallback 流程**可操作

**未达标的表现**：三层 eval 只是概念复述 / trace 和 eval 还是两套 / Judge 不校准 / Flywheel 有环但没人负责。

## 贯穿项目挂接点

Unit 2 让 [🛠️ SRE 事故助手](../贯穿项目-SRE事故助手.md) 从"自称能用"变成"有度量证明可用"：

| Week | 项目新增 | 验收 |
|---|---|---|
| W1 | `app/trace.py` + `traces/` — trace schema + JSONL 落盘 | 每次请求完整可回放 |
| W2 | `app/eval.py` + `data/eval/gold.jsonl`（≥30 条）— L1 assertion + L2 judge | 能跑一轮完整 eval 报告 |
| W3 | Judge 对齐度校准：10-20 条人工 + judge 打分对比 | 记录 Cohen's κ 或 agreement rate |
| W4 | `app/failure_queue.py` + `docs/data-flywheel.md` | 失败样本入队；每步 owner 明确 |

**走完 Unit 2，你的项目**：每次响应有 trace、有质量分、失败样本能回流。这是 Unit 3 做 SLO 的前提——没有度量就没有 SLO。
