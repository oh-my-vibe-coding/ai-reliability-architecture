# 深入 01 · 影响首包延迟（TTFT）与 Token 吞吐速度的因素

> [← 返回目录](../README.md)  ·  对应知识章节：[第 5 章 · AI 推理服务的可靠性工程](../知识/05-AI推理服务的可靠性工程.md)

给 SRE 架构师用的工程级解析。如果你要为 LLM 推理服务做 SLO / 容量规划 / 延迟优化，这篇是基础。

这一节用一个比喻+一个具体场景把直觉搭好，再正式进 prefill/decode 的机制——**不需要任何 GPU 或 ML 背景**就能看懂前面两节。

---

## 0. 一个比喻：去餐厅吃饭，菜不是一次端上来

想象你去一家私厨点一份套餐：

1. 你把菜单和备注（"这个少辣、那个不要香菜"）递过去
2. 厨师**先把整张菜单从头读完**——理解你点了什么、有什么口味要求
3. 然后他开始做并**分次上菜**：先一碗汤、再凉菜、再主菜、再甜点
4. **每上一道菜之前**，他都要回到厨房重新看一遍：你点了什么 + 之前已经上过什么 + 接下来应该是什么

LLM 推理就是这两个阶段：

- **"读菜单"阶段** = **Prefill**：把你的 prompt 一口气读完，在内部建好理解。**TTFT（首包延迟，Time To First Token）就是这一步耗了多久**。
- **"分次上菜"阶段** = **Decode**：一个 token 一个 token 往外吐。**tokens/s（每秒输出几个 token）就是这一步的速度**。

这两个阶段的瓶颈**完全不同**——这是这一节最重要的事实：

| 阶段 | 卡在哪儿 | 一句话直觉 |
|---|---|---|
| 读菜单（Prefill） | **算力** | 菜单越长读得越久，但只读一次 |
| 分次上菜（Decode） | **显存带宽** | 每上一道菜都要回去翻菜单 + 看前面上过的菜，反复读取 |

为什么 SRE 必须把这两个分开看？因为它们的**优化手段、容量边界、监控指标全都不同**。把它们合在一起当"总延迟"看，等于把厨房和上菜服务员的问题搅在一起——永远找不到根因。

---

## 1. 一个具体场景：为什么"总延迟一样"两家服务体验差很远

你切换了两家 LLM 服务跑同一个聊天产品，**平均总延迟都是 8 秒**。但客户反馈完全不同：

- **A 厂**：「要等好久才开始有回应，但出来之后字儿飞快。」
- **B 厂**：「立刻就开始打字了，但打字慢得像在思考人生。」

如果只看"总延迟 = 8 秒"这一个数，你看不到这两种体验差异——而它们对用户感知是天差地别的（产品同事会告诉你 B 厂体验**完全可以接受**，A 厂用户**直接放弃了**）。

它们对应的根因也完全不同：

- A 厂慢在 TTFT（prefill 慢）：可能上的是大模型 + 没开 caching、可能有几万 token 的长系统提示、可能上游排队
- B 厂慢在 tokens/s（decode 慢）：可能 GPU 带宽不够、可能 batch 排得太满、可能模型量化激进导致每步耗时增加

> **这就是为什么 LLM 服务的 SLO 必须把 TTFT 和 tokens/s 分开定**——只盯一个总延迟，等于不同病用同一种药。

把这两个分开之后，剩下的全部内容就是回答两个问题：
- **TTFT 受什么影响？**（§3）
- **tokens/s 受什么影响？**（§4）

## 2. 正式拆解：Prefill 阶段 vs Decode 阶段

现在把比喻替换成真实的计算流程。这一节把术语都"挂回"前面的直觉。

| 阶段 | 处理对象 | 计算特性 | 决定 |
|---|---|---|---|
| **Prefill** | 一次性把全部输入 token 喂给模型，算出最后一个位置的输出分布 | **Compute-bound**（算力瓶颈） | **TTFT** |
| **Decode** | 一次生成一个 token，循环 N 次 | **Memory-bandwidth-bound**（显存带宽瓶颈） | **tokens/s**（每秒输出 token 数） |

**为什么 Prefill 是算力瓶颈**？输入有 N 个 token，模型可以**同时**对它们做一次大矩阵乘法（"一口气把菜单读完"），GPU 算力被压满。计算量大约是 `O(N² × d)`（d 是模型隐藏维度），N 越大越慢，但 GPU 跑得很满，**算力是天花板**。

**为什么 Decode 是带宽瓶颈**？每生成一个 token，要做一件听起来很奇怪的事——**把模型全部参数从显存读一遍**（70B 模型 ≈ 140 GB），还要把当前序列的全部 KV cache 也读一遍，然后只算"下一个 token 是什么"。算力用得很少，但**显存带宽被打满**。带宽是天花板。

#### 两阶段数据流

```
┌─ PREFILL（读菜单） ────────────────────┐
│                                        │
│  输入 [t1, t2, ..., tN]                │
│         │                              │
│         ▼                              │
│   一次性算所有 Q, K, V  ← 大矩阵乘     │
│         │                              │
│         ▼                              │
│   算出第 N+1 个位置的 logits           │
│         │                              │
│         └─► 采样第一个输出 token       │
│                                        │
│  同时：存下 K 和 V → KV Cache         │
│                                        │
│  瓶颈：GPU FLOPs (compute-bound)       │
│  成本：~ O(N² × d)                     │
└────────────────────────────────────────┘
                 │
                 ▼
┌─ DECODE（一道一道上菜） ────────────────┐
│                                        │
│  对于每一个新 token（重复 M 次）：     │
│                                        │
│     读全部模型参数（140GB for 70B）    │
│     + 读当前 KV Cache                   │
│              │                         │
│              ▼                         │
│     只算当前位置的 Q                   │
│     和历史所有 K, V 做 attention       │
│              │                         │
│              ▼                         │
│     采样 → 输出一个 token              │
│     并追加到 KV Cache                  │
│                                        │
│  瓶颈：HBM 带宽 (memory-bound)         │
│  成本：~ O(每 token) × M              │
└────────────────────────────────────────┘
```

**两个最实用的工程推论**（直接照抄到设计文档里）：
- 长输入、短输出场景（如文档总结）→ **TTFT 是主要延迟**，优化重点在 prefill
- 短输入、长输出场景（如代码生成、Agent 多步规划）→ **tokens/s 是主要延迟**，优化重点在 decode

## 3. 影响 TTFT（首包延迟）的因素

TTFT = `从你发出请求到收到第一个 token` 的时间。它可以进一步分解：

```
TTFT = 网络 RTT
     + 客户端到 Gateway 的排队
     + Gateway → Inference Server 的路由
     + Inference Server 上的排队（等待 batch 窗口）
     + Tokenization
     + Prefill 计算（占大头）
     + 首 token 采样
     + 首 token 流式返回的网络传输
     + 客户端 SDK 的缓冲 flush
```

下面挨个看哪些因素影响哪一段。**Prefill 计算那一段（第 6 行）最大也最值得动手**——所以从它开始。

### 3.1 输入 token 数量（最主要因素）

Prefill 的计算量**大致与输入 token 数线性相关**（近似 `FLOPs ≈ 2 × N × P`，N 是输入 token 数，P 是参数量）。

- 输入从 1k → 10k token，prefill 时间约增加 10×。
- 长 context 场景（128k token）下，prefill 可能需要数秒。

**工程含义**：如果你的场景输入长（长文档、长系统提示），TTFT 永远不会低。要么缩短输入，要么上 prompt caching（见[深入 02](02-Prompt-Caching原理.md)）。

### 3.2 模型大小（参数量和精度）

- 参数量：7B vs 70B，prefill 耗时近似差一个数量级。
- 精度：fp16/bf16 是主流；int8/int4 量化可以把 prefill 时间降 2-4×，但有精度损失风险（见第 5 章静默降级）。

### 3.3 硬件

| 硬件特性 | 对 TTFT 的影响 |
|---|---|
| GPU 算力（TFLOPS） | 决定 prefill 的理论上限 |
| GPU 显存带宽（HBM bandwidth） | Prefill 的内层循环还是需要带宽，但影响小于 decode |
| GPU 数量（TP/PP 并行度） | 增加并行度能降低大模型 prefill 时间，但有通信开销 |
| InfiniBand / NVLink | 多卡之间的通信延迟，大模型 TP 场景下显著影响 |

**常见 GPU 档位**（2026-06 快照 🕒，关键数字记下来。全书 GPU 参数以这张表为唯一来源）：

| GPU | BF16 算力（dense） | HBM 带宽 | 显存 | 一句话定位 |
|---|---|---|---|---|
| H100 SXM | ~989 TFLOPS | 3.35 TB/s | 80 GB | 算力/带宽都强的通用基准 |
| H200 SXM | ~989 TFLOPS | 4.8 TB/s | 141 GB | **算力同 H100，带宽+显存涨** → 主打 decode 与大模型 |
| A100 | ~312 TFLOPS | 2 TB/s | 40/80 GB | 上一代主力 |
| L40S | ~362 TFLOPS | 864 GB/s | 48 GB | 带宽低，decode 慢 |
| **H20**（中国特供 🇨🇳） | **~148 TFLOPS** | **4.0 TB/s** | 96 GB | **算力只有 H100 的 ~1/6.7，带宽反而更高** |

> [!IMPORTANT]
> 记住 H20 这个反差——**算力被出口管制砍到 H100 的约 1/6.7，带宽却比 H100 还高**。它把"prefill 卡算力、decode 卡带宽"这条原理逼到了极端：同一张卡，两个阶段的体验会彻底分裂。§3.3 末尾和 §4.2 各算一半，§5 合起来算总账。

#### 算一笔：H20 上部署 Qwen2.5-32B，prefill 慢多少

拿一个国内常见组合——在 **H20 上自建 Qwen2.5-32B**（dense），输入 8k token（一段不算长的长文档 + 系统提示）。用 §3.1 的 `FLOPs ≈ 2 × N × P`：

```text
Prefill FLOPs ≈ 2 × 8000 × 32e9 = 5.12e14 = 512 TFLOP
```

除以各卡算力，得**理论下限**（满 MFU，实际按 MFU≈0.4 再翻一倍多，但卡之间的**比值不变**）：

- H100：512 / 989 ≈ **0.52 s**
- H20：512 / 148 ≈ **3.5 s**

**H20 的 prefill 比 H100 慢约 6.7×**（就是算力比 989/148）。注意此处 H20 带宽（4.0）比 H100（3.35）还高——**但 prefill 卡的是算力，带宽再高也救不了 TTFT**。这就是为什么长输入场景在 H20 上体验会很像 §1 的「A 厂」：字出来之后不慢，但要等好久才开始。

### 3.4 Batching 策略

现代推理服务（vLLM / TGI / SGLang）普遍用 **Continuous Batching**：
- 多个请求共享一次 GPU 调用，提高吞吐。
- **副作用**：单个请求必须等到 batch 窗口凑够才能被处理，引入排队延迟。

**权衡**：
- Batch size 越大，整体吞吐越高，**但单请求 TTFT 越不稳定**。
- P50 可能很好，但 P99/P99.9 会被 batching 冲突放大。

### 3.5 系统负载与排队

当并发请求数超过 `max_batch_size × concurrency` 时，请求会进入排队。排队时间在高负载下可能**远超** prefill 本身。

**工程指标**：
- **KV cache 占用率**：一个实例能容纳的并发请求数由显存决定（每个活跃请求占 KV cache）。占用率逼近 100% 时，新请求被拒绝或排队。
- **Waiting queue 长度**：vLLM 等都暴露了这个指标，必须作为 SLI 监控。

### 3.6 冷启动（Cold Start）

如果你的推理服务是按需启动（serverless 或 autoscaler 场景）：
- 模型权重装载：70B bf16 = 140 GB，从磁盘/网络加载可能需要 30s–数分钟
- CUDA 初始化、warmup 推理：数秒到数十秒
- JIT 编译（XLA / Triton kernel）：首次请求显著慢

**对抗方式**：预热实例、保留 minimum replicas、模型权重 pin 在共享存储。

### 3.7 Tokenization 开销

对超长输入（>100k token），tokenizer 本身可能需要几十毫秒到上百毫秒。通常可以忽略，但在边缘场景会被注意到。

### 3.8 Safety Filter / 前置处理

托管 API（如 OpenAI Moderation、Anthropic 的 safety layer）会在 prefill 之前或之后做一层检查，增加几十毫秒到数百毫秒。

### 3.9 网络与 SDK

- 客户端到服务的 RTT：跨地域连接会显著增加延迟
- SDK 的流式实现是否有缓冲：Python `httpx` 默认的 chunk 大小、buffer flush 策略都会影响**你观测到的** TTFT
- HTTP/2 vs HTTP/1.1：流式场景 HTTP/2 更好

### 3.10 网关位的 TTFT 四段归因（Gateway-side Attribution）

前面 3.1–3.9 列的 9 段拼起来才是用户感知 TTFT，但**真正能在生产里把它们拆开归因的位置只有一个：LLM 网关**。单独一台推理服务、或者单独一份客户端 SDK，都看不全。

把开篇那张分解图重新归类，对**网关运维方**而言只剩四段：

```
TTFT_user
  ≈ TTFT_client       # 客户端发出 → 网关收到首字节请求
  + TTFT_gateway      # 网关排队 / 鉴权 / 路由 / 通道选择
  + TTFT_upstream     # 网关 → 上游推理服务（含跨境 RTT）
  + TTFT_model        # 上游内部排队 + Prefill + 首 token 采样
```

**为什么必须按这四段建 SLI**：

- 每一段的 owner 不同：客户端段是调用方的事，网关段是平台 SRE 的事，上游段是网络/供应商的事，模型段是推理团队的事。**不分段的 TTFT 告警没法分派**。
- 每一段的优化手段不同：网关段堵了上 inflight 限流（见[深入 05](05-LLM推理服务的容量规划.md)），上游段堵了切通道，模型段堵了上 caching（见[深入 02](02-Prompt-Caching原理.md)）。
- **同一个 P99 退化，可能由完全不同的段贡献**——只看总 TTFT 一定误判。

**网关位至少要落的字段**（一次请求一行日志，每段毫秒）：

| 字段 | 含义 | 数据来源 |
|---|---|---|
| `ttft_gateway_ms` | 收到请求 → 真正开始向上游发包 | 网关自己埋点 |
| `ttft_upstream_ms` | 上游连接建立 → 首字节回包 | 网关 HTTP client 钩子 |
| `ttft_model_ms` | 上游首字节 → 真正首 token（剥掉协议头/keepalive 字节） | 流解析时第一个 token chunk 的时间戳 |
| `ttft_total_ms` | 三者之和（≈ 用户感知 - 客户端段） | 派生 |

**易踩的坑**：

- 把 SSE 协议的 `: ping` 心跳/空白行当成"首字节"——`ttft_upstream_ms` 会被严重低估。**首字节判定必须以"出现可解析为 token 增量的事件"为准**。
- 用网关入口时间戳减客户端时间戳算 `ttft_client_ms`——客户端时钟不可信。这一段只能靠客户端 SDK 自己上报，网关侧能给出的最准的"用户感知 TTFT"近似就是 `ttft_total_ms`。
- 拿单条请求的四段做归因。**一定要在分组（model × upstream × route_tag）下看分位数**，单条会被异常 batching 干扰。

**实用推论**：

- 没有这四段拆解的 SLO 不是 LLM SLO。`p99 TTFT < 2s` 这种总指标在多上游网关里只能当"健康灯"，不能当"诊断工具"。
- 网关一旦把这四段暴露给上层 dashboard，**调用方就再也回不去"只能看 latency_ms"的世界**——这是平台对调用方最实在的价值之一。

## 4. 影响 Token 输出速度（tokens/s）的因素

Decode 阶段的 tokens/s 是另一套约束。核心结论：

> **Decode 是显存带宽受限的**。优化带宽利用，就是优化 tokens/s。

回想 §0 的比喻：上每一道菜都要回厨房翻一遍菜单+前面已经上过的菜——**带宽就是"厨师从厨房到菜单的来回速度"**。

### 4.1 显存带宽是第一瓶颈

Decode 每生成一个 token，需要：
- 把**全部模型参数**从 HBM 读到计算单元（70B bf16 ≈ 140 GB）
- 把**当前序列的全部 KV cache** 从 HBM 读进来

用 HBM 带宽算 **理论上限**：
```
理论 tokens/s ≤ HBM 带宽 / 每 token 需读取的数据量
```

**例：70B bf16 on H100**
- HBM 带宽：3.35 TB/s
- 每 token 读取量：~140 GB（参数）+ KV cache（短序列下可忽略）
- 理论上限：3.35 / 140 ≈ 24 tokens/s
- 实际（单请求、无优化）：15–20 tokens/s

这个数学是**硬约束**。没有算法魔法能超过它。想快就只能：减少每步读取量（量化、MoE）、增加带宽（更好硬件）、或改变 decode 方式（speculative、并行 decoding）。

### 4.2 模型大小（直接影响带宽需求）

- 7B bf16 = 14 GB → 单 H100 理论上限 ~240 tokens/s
- 70B bf16 = 140 GB → 单 H100 ~24 tokens/s
- 405B bf16 = 810 GB → 单 H100 装不下，需要 TP 分布到多卡（**怎么分、token 怎么保，见 [深入 20](20-单卡装不下的大模型分布式推理.md)**）

**再算一笔：还是那台 H20 上的 Qwen2.5-32B**（32B bf16 ≈ 64 GB），换 decode 视角。每生成一个 token 要把 64 GB 权重读一遍，理论上限 = 带宽 / 64：

- H100：3350 / 64 ≈ **52 tokens/s**
- H200：4800 / 64 ≈ **75 tokens/s**
- H20：4000 / 64 ≈ **62 tokens/s**

**H20 的 decode 反而比 H100 快**（带宽比 4.0/3.35 ≈ 1.19×）。同一张 H20、同一个模型——§3.3 里它 prefill 慢到 6.7×，这里 decode 却反超 H100。这不是矛盾，正是「prefill 卡算力、decode 卡带宽」两套瓶颈在一张卡上的真实分裂：**H20 拿来跑短输入、长输出（decode 重）很划算，拿来啃长文档（prefill 重）就是灾难**。§5 把这两半合起来算一次总账。

### 4.3 Quantization（量化）

把权重从 bf16（2 字节/参数）压到 int8（1 字节）或 int4（0.5 字节），**带宽需求直接减半或更多**：
- int8 70B ≈ 70 GB → 理论 ~48 tokens/s
- int4 70B ≈ 35 GB → 理论 ~95 tokens/s

**代价**：精度损失。SRE 要和 eval 团队配合验证量化后是否有质量回退（这就是第 5 章静默降级提到的）。

### 4.4 Batch Size（对 tokens/s 影响复杂）

- **单请求视角**：batch 越大，单请求的 tokens/s **不会变快**，甚至因为 scheduling 可能更慢。
- **总吞吐视角**：batch 越大，**总** tokens/s（所有请求合计）越高——因为模型权重只读一次，能服务多个请求。

这是为什么推理服务优化总吞吐（throughput）和优化延迟（latency）会冲突。

**工程含义**：用户感知的速度 ≈ 单请求 tokens/s。服务成本 ≈ 总吞吐的倒数。**两者无法同时最优**，必须做 tradeoff。

### 4.5 KV Cache 的"拖后腿"效应

随着输出变长，当前序列的 KV cache 会线性增长。**每生成一个 token，都要把全部 KV cache 读一遍**。

- 短输出（< 几百 token）：KV cache 占带宽的比例小
- 长输出（10k+ token）：KV cache 可能比权重还大，每步都要读，tokens/s 显著下降

**观测实验**：同一个模型，输出 100 token 和输出 10000 token，后者的**平均 tokens/s 会明显低于前者**。这不是 bug，是架构特性。

### 4.6 Speculative Decoding（投机解码）

用一个小模型先"猜"几个 token，再用大模型一次性验证。如果猜对，一次带宽读取吐出多个 token。

- 理论加速 2–4×，实际取决于接受率。
- 对 SRE：这会让 tokens/s 的分布从"平稳"变成"阵发"（一次吐 3 个或只吐 1 个），监控指标需要调整。

### 4.7 并行策略

| 策略 | 作用 | 延迟影响 |
|---|---|---|
| **TP (Tensor Parallel)** | 把单层算子切到多卡 | 降低单 token 延迟（单 token 算得更快），但增加卡间通信 |
| **PP (Pipeline Parallel)** | 把不同层放到不同卡 | 降低显存压力，但引入 bubble（流水线气泡）可能增加 TTFT |
| **EP (Expert Parallel)** | MoE 场景，把 expert 分到不同卡 | MoE 专属，能降低每 token 读取量 |

TP 对 decode 有直接加速，但 2 卡翻倍、4 卡不到 4 倍（通信开销不是线性的）。**这三种并行怎么组合、为什么 TP 只能待在一台机器里而 PP/EP 能跨节点，见 [深入 20 · 分布式推理](20-单卡装不下的大模型分布式推理.md)。**

### 4.8 硬件的 decode 专门化

部分新硬件专门为 decode 优化：
- B200、H200 主打更高带宽
- MI300X 单卡 192 GB 显存，能塞更大模型
- Groq LPU、Cerebras WSE：专用架构，声称 tokens/s 极高（也有 context 限制）

## 5. 综合算例：同一张 H20，为何 decode 香、prefill 苦

§3.3 和 §4.2 各算了一半，这里合起来看一次总账——这也是 §1 那个「A 厂 vs B 厂」故事的真实硬件版：这次两种体验来自**同一张卡**，只因负载形态不同。

### 5.1 Dense 模型：Qwen2.5-32B 在三张卡上的分裂

同一个 Qwen2.5-32B（bf16 ≈ 64 GB），8k 输入、流式输出，三张卡的两个阶段：

| 阶段（瓶颈） | H100 | H200 | H20 | H20 相对 H100 |
|---|---|---|---|---|
| **Prefill / TTFT**（卡算力） | ~0.52 s | ~0.52 s | ~3.5 s | **慢 6.7×** 😖 |
| **Decode / tokens/s**（卡带宽） | ~52 | ~75 | ~62 | **快 1.19×** 😀 |

> 数字均为理论下限（满 MFU、KV cache 忽略），实际会打折，但**卡之间的比值稳定**。prefill 用 §3.1 的 `2NP`，decode 用 §4.1 的 `带宽 / 每步读取量`。

一张表把 §0 的核心事实钉死了：**prefill 和 decode 卡在完全不同的资源上，所以同一张卡可以在一个阶段是最差的、在另一个阶段是最好的。** 选卡不能问"H20 好不好"，只能问"我的负载是 prefill 重还是 decode 重"——直接对应 §2 末尾那两条工程推论：

- 长输入短输出（文档总结、RAG 大 context）→ **prefill 重** → H20 是灾难，该上 H100/H200 或开 caching
- 短输入长输出（代码生成、Agent 多步、长文创作）→ **decode 重** → H20 的高带宽反而让它性价比可能反超

### 5.2 MoE 模型：DeepSeek-V3/R1 为什么能"在带宽上作弊"

换 DeepSeek-V3/R1（MoE，671B 总参数、每 token 只激活 **37B**）。MoE 对这一章的两个瓶颈做了不对称的事：

- **Decode（带宽）省了**：每生成一个 token 只需读被激活的那 ~37B 参数，不是全部 671B。bf16 下每步读 ~74 GB → H20 上 4000/74 ≈ **54 tokens/s**。一个 671B 的大模型，decode 速度却像个 37B 的小模型——这就是 MoE「在带宽上作弊」的地方。
- **显存（容量）没省，反而更凶**：路由到哪个专家是逐 token 变的，所以**全部 671B 专家必须常驻显存**。bf16 ≈ 1342 GB、fp8 ≈ 671 GB。H20 单卡 96 GB → fp8 至少 **~7 张卡**、bf16 **~14 张**才装得下权重（还没算 KV cache）。

于是 MoE 天然把你推向 §4.7 的 **EP（Expert Parallel）+ TP**：专家摊到多卡，跨卡通信（NVLink/IB）就成了新的延迟来源。**这台"单卡装不下"的机器到底怎么被拼出来、token 怎么保住，见 [深入 20 · 分布式推理](20-单卡装不下的大模型分布式推理.md)——它用 DeepSeek-671B 做了完整的端到端算例。**

> [!NOTE]
> 一句话记住 MoE：**它把 decode 的午餐钱，记到了显存和跨卡通信这另一张账单上。** 省下的带宽是真的，但没有免费的午餐——只是账单换了个地方寄。

> [!TIP]
> **H20 选型直觉**（2026-06 快照）：短输入长输出、decode 重的场景（Agent、代码、长文生成），H20 的高带宽让它性价比可能反超 H100；长输入短输出、prefill 重的场景（长文档总结、大 context RAG），别硬上 H20，要么换算力强的卡、要么先上 prompt caching（[深入 02](02-Prompt-Caching原理.md)）把 prefill 摊掉。

> [!WARNING] 何时这条不再适用
> 上面的"H20 跑 decode 划算"结论建立在**出口管制造成的算力/带宽畸形配比**之上。当以下任一事件发生，请重算而不是照抄：
> - 出口管制放开或换代，H20 与全球版的价差/规格差消失 → 性价比逻辑失效；
> - 你的业务从"短输入长输出"漂移到"长 context"（比如接入了大文档 RAG）→ prefill 重新变成主瓶颈；
> - 推理范式离开自回归逐 token 解码（如扩散式 LLM 成熟）→ "decode 卡带宽"这个前提本身不再成立。

## 6. 尾延迟（P99 / P99.9）

P50 看起来好，P99 往往吓人。原因：

1. **Batching 冲突**：请求运气不好，落在满 batch 的尾巴上，被迫等下一轮
2. **KV cache preemption**：当显存快满时，服务可能驱逐长序列请求的 cache，重算代价巨大
3. **GC / JIT**：偶发的编译或内存整理
4. **Hardware hiccup**：ECC 错误、thermal throttling
5. **Network tail**：偶发丢包、重传
6. **Cold cache / first request**：见 §3.6

**SRE 的监控重点**：
- TTFT p50 / p95 / p99 / p99.9 **分别**监控
- tokens/s p50 / p95 分别监控
- 队列长度、KV cache 占用率、preemption 次数

## 7. 对 SRE 的 SLO 设计含义

### 为什么 TTFT 和 tokens/s 必须分开定 SLO

回到 §1 那个故事：A 厂和 B 厂"总延迟"一样但用户体验天差地别。原因就是：

- 它们**机制不同**（prefill vs decode）
- 它们**优化手段不同**
- 它们**对用户体验的意义不同**：
  - TTFT 决定"感觉得到不到等待"
  - tokens/s 决定"看起来流畅不流畅"

### 推荐 SLI 集合（至少这些）

| SLI | 说明 | 目标参考 |
|---|---|---|
| TTFT p99 | 首 token 延迟 99 分位 | 对话类 <1s，长 prompt 类 <5s |
| Output tokens/s p50 | 输出速度中位数 | 取决于模型，15–80 tokens/s |
| Request success rate | 不是 timeout / OOM | > 99.5% |
| Queue depth | 排队深度 | < batch_size × 2 |
| KV cache utilization | 显存占比 | < 85%（留 preemption 缓冲） |
| Preemption rate | 抢占率 | < 1% |
| End-to-end latency | 整体延迟（不是 TTFT） | 和业务目标对齐 |

### 容量规划的起点

单实例极限 ≈ min(
- HBM / (模型大小 + 平均 KV cache per request)，
- Prefill 算力 / 平均输入 token 数,
- 总带宽 / 平均输出 token 数
)

不会有一个公式给你精确答案，但这三个上界都要算，取最小。详细推导见[深入 05](05-LLM推理服务的容量规划.md)。

## 8. 如果你要排查"突然变慢"

按这个顺序查：

1. **请求队列深度**：是不是并发突然涨了？
2. **KV cache 占用率**：有没有抢占？
3. **单请求输入长度分布**：有没有变长？（最常见的"莫名其妙变慢"原因）
4. **Batch size / 并发请求数**：调度策略是否变化？
5. **硬件指标**：GPU 温度、power throttling、ECC errors
6. **上游**：tokenizer 更新？safety filter 更新？
7. **模型版本**：是不是灰度切流量了？
8. **网络**：client 到 server 的 RTT 有没有变？
