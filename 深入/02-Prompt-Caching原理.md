## 0. 一个比喻：每天对失忆学生重讲开场

想象你每天给同一个学生上 1 小时课。这个学生**每天都失忆**：每次见面，他都不记得你是谁、上节课讲到哪、课程目标是什么。

所以你的每一节课都长这样：

```text
[ 30 分钟开场 ]
  - 我是谁，你是谁
  - 我们这门课的目标
  - 之前 5 节课讲过的关键点
  - 今天可以用到哪些参考资料

[ 30 分钟新内容 ]
  - 今天真正要讲的东西
```

每天前 30 分钟讲的话**几乎一字不差**，但你不得不每天讲一遍——因为他不记得。

LLM 就是这个学生。模型本身没有跨调用的记忆，每次 API 请求都必须把"开场那一大段"重新塞进 prompt。这一大段在工程上很贵（见[深入 04 · 为什么简单"你好"也消耗数万 Token](04-为什么简单你好也消耗数万token.md)）。

现在注意一个容易忽略的细节：学生听开场的时候，手里的笔记本上会留下**他消化之后的理解**——不是你讲话的逐字稿，而是他脑子里建立起来的那份索引。

> **Prompt Caching 做的事**：下课后把这本笔记收走存档。第二天，只要你的开场一字不差，就把昨天的笔记直接放回他面前——他的大脑瞬间恢复到"昨天听完开场那一刻"的状态，你从第 31 分钟开始讲新内容。

为什么存的是"笔记"而不是"录音"？录音里是你讲的话——那就是 prompt 本身，每次请求本来就带着，不缺这份；把录音再放一遍，学生还是得从头听一遍才能进脑子。真正贵的是"**听懂**"这个过程（工程上叫 prefill，把输入一口气读完、在内部建立理解——[深入 01](01-首包延迟与吞吐的影响因素.md) 的主角）。所以缓存必须存"听懂之后的状态"。这个区分不是修辞，它就是 §2 的全部内容。

§1 先用一笔真实账单告诉你这件事值多少钱；之后的主体小节回答四个朴素的问题：**笔记里记的是什么（§2）/ 怎么判定开场一字不差（§3）/ 笔记保存多久、什么时候作废（§4、§5）/ 作废了 SRE 怎么办（§6）**。

## 1. 一个具体场景：RAG 客服的账单为什么差 8 倍

你做了一个客服助手，工作模式是：

```text
[ 系统提示 5k token ：你是 X 公司的客服 ... ]
[ 知识库节选 30k token ：产品手册前 60 页 ... ]
[ 用户问题 200 token ：我的订单为什么还没发货？ ]
```

每个用户的问题不同，但前面 35k token 是**完全一样**的（同一个知识库节选）。

不开 caching 时，每次请求服务都要：
- 把全部 35,200 token 喂进模型，从头算一遍——这一步就是 prefill
- 然后才开始生成回答

开 caching 时：
- 第 1 次请求：照常算完整 35,200 token，但顺手把前 35,000 token 的"中间状态"存下来
- 第 2 次开始：前 35,000 token 跳过计算，**直接读取存下来的中间状态**，只对最后 200 token 做 prefill
- 计算量从 35,200 → 200，**降到约 1/175**

按 Anthropic 当前定价折算成钱：
- 命中缓存的 token：**0.1×** 常规价（省 90%）
- 写入缓存的 token：**1.25×** 常规价（多付 25%，因为要存一份）
- 没缓存的 token：常规价

注意两个口径细节：

- **0.1× 是托管 API 的计费折扣，不是物理成本**。物理上省掉的计算是 175×——厂商没有把省下的算力全让给你（自建推理时两本账要分开算，见[深入 05](05-LLM推理服务的容量规划.md)）。
- **缓存只作用于输入侧**。输出 token 一分不打折——输出重的场景，缓存对总账单的杠杆会被稀释。

如果同一份知识库一天被命中 1000 次，第 1 次贵了 25%，后 999 次每次输入费省近 90%——摊下来，输入侧账单大约下降 **89%**：前缀占了输入的 99% 以上，第 1 次的写入溢价摊到近千次命中里可以忽略。再算上一分不打折的输出，总账单的差距大致就是标题里那个 8 倍。

这就是为什么这个东西值得一整节去讲。

## 2. 笔记里记的是什么：不是回答，是"读完之后的状态"

很多人第一反应是："缓存的是问题对应的答案吧？"——**错了**。

如果缓存的是答案，那就解释不了：
- 为什么 `temperature > 0` 时同一个 prompt 命中缓存，回答还会不一样？
- 为什么换一个用户问题，前面的缓存还能复用？

真相：缓存的是**模型读完前面那段话之后，"脑子里"形成的中间状态**。这个中间状态叫 **KV cache**——Transformer 每一层、每个注意力头算出的 K（Key）与 V（Value）张量。之所以能缓存，是因为一个物理事实：**只要前文不变，历史 token 的 K/V 永远不变**（为什么不变，见[科学 01 · Attention 的 SRE 视角](../科学/01-Attention与Transformer的SRE视角.md)）。用一句话对齐 §0 的比喻：

> 模型读了 35k token 的产品手册之后，在"工作记忆"里整理出了一份索引。Prompt Caching 把这份索引（KV cache）存下来。下一次同样的 35k token 进来，不用重读手册、重建索引，直接把索引放回去接着用。Prompt Caching 就是**把 KV cache 的生命周期从"单次请求"延长到"跨请求"**。

**三个关键事实**（不需要背，知道有这回事就行）：

- 这份"笔记"**比原文大四五个数量级（几万到十万倍）**：1k token 的原文只是几 KB 文本，但 70B 级模型（GQA 注意力结构 + fp16 精度，主流 70B 的典型配置）的 KV cache 约 0.3 MB/token——1k token 对应**几百 MB**（精确公式与逐模型数字见[深入 05 · 容量规划](05-LLM推理服务的容量规划.md)）。存的是"理解"，不是"文本"，理解本来就比文本占地方。
- 所以缓存**占显存**，会和"能服务多少并发"互相挤占（§6.3）。托管厂商与自建生态都在把它往外搬——GPU 显存放不下就下沉到 CPU 内存、本地 SSD 甚至远端存储，**KV cache 正在从"单实例里的临时状态"变成"集群级存储资源"**（§9 有这条生态线的入口）。
- 笔记**只对当前模型有效**：换模型版本、换量化方式，笔记全部作废（§5.6）。

## 3. 怎么判定"一字不差"：Token 级前缀匹配

这是最多人踩坑的地方。**"一字不差"不是指字符串相等，是指 token 序列的前缀相等**。

### 3.1 为什么是 token，不是字符

模型不直接处理字符，先把字符切成 token。判定缓存是否命中，**比的是切完之后的 token 序列**。两段"看起来一样"的文本，token 序列可以不一样：

- `"Hello world"` 和 `"Hello  world"`（两个空格）切出来不同
- 「效果不错👍」和「效果不错 👍」——emoji 前多一个空格，切法就变
- 中英混排里，全角/半角标点（`，` vs `,`）是不同的 token

Tokenizer 的这些坑有专门一章：[科学 04 · Tokenization 的坑](../科学/04-Tokenization的坑.md)。

### 3.2 为什么是"前缀"，不是任意位置

模型读 prompt 是**严格从左到右**的。第 K 个 token 的中间状态，依赖前面 1~K-1 个 token——回到 §0 的比喻：笔记是逐句消化出来的，第 10 句的理解叠在前 9 句之上；你把第 2 句换了个词，从第 2 句起的笔记全都不再可信，只能从那里重记。

所以：
- ✅ 「[A][B][C][D]」的缓存可以给「[A][B][C][E]」复用前 3 个 token 的部分（前缀 ABC 一样）
- ❌ 不能复用给「[X][B][C][D]」（第一个 token 就不同，后面全部重算）
- ❌ 不能复用给「[A][X][C][D]」（第二个 token 一改，C/D 的中间状态都得重算）

### 3.3 工程上的一句话规则

> **越靠前的内容应该越稳定，会变化的内容尽量往后放。**

把这句话翻译成具体不该做的事：
- ❌ 系统提示开头放当前时间戳 → 每次请求时间戳都变，整个缓存全废
- ❌ 系统提示开头放用户 ID → 每个用户独享一份缓存，复用率从「按业务级」降到「按用户级」（不一定是 bug，但要算清楚）
- ❌ 系统提示开头加版本标签做 A/B 测试 → 切版本时缓存全部失效

应该这样：
- ✅ 固定的系统提示放最前
- ✅ 长期不变的参考文档接在后面
- ✅ 会变的当前对话、当前问题放最后

### 3.4 "前缀"不只是消息文本

参与匹配的是**整个请求的可缓存部分**，不只是你肉眼看到的那段话：

- **工具定义**（函数名、描述、参数 schema）在系统提示之前参与匹配——**改一行工具描述 = 前缀从头变**。Agent 产品迭代时 prompt 没动、只改了工具注释，缓存照样全灭，这是 2026 年最常见的"命中率无故崩了"根因之一
- **结构化输出的 schema、图片**也在匹配范围内（OpenAI 连图片的 `detail` 参数都要求一致）
- **序列化顺序**也算数：部分语言的 JSON 库（Swift、Go）默认 key 顺序不稳定，同一个请求体两次序列化字节不同——缓存静默 miss，日志里什么都看不出来

### 3.5 实战判定：别靠肉眼，靠回放和诊断

`"hello world"` 和 `"hello world "`（尾部多一个空格）在很多模型上分词结果就不同。**别假设"看起来一样"就 OK**：

- 上线前用**真实流量回放**对一遍命中率
- Anthropic 有 beta 的 Cache diagnostics：API 对比相邻两次请求，直接报告前缀从哪个位置开始分叉——把"diff 渲染后的 prompt 字节"这个手工活自动化了

## 4. 三家厂商怎么做：同一个机制，三种产品化（2026-07 快照 🕒）

**一句话记住三家**：Anthropic——你告诉它"这段帮我记住"；OpenAI——它自己判断要不要记；Gemini——默认自己记，你也可以付"租金"要求它保证记住。机制原理相通，但控制粒度、计费模型、保留时长差别很大。

### 4.1 Anthropic：客户端显式打标记

你在请求里告诉它"这一段我希望缓存"：

```json
{
  "model": "claude-sonnet-4-6",
  "system": [
    {
      "type": "text",
      "text": "你是一个 SRE 助手...（长系统提示）",
      "cache_control": {"type": "ephemeral"}
    }
  ],
  "messages": [
    {
      "role": "user",
      "content": [
        {
          "type": "text",
          "text": "参考文档：...（几万 token 的文档）",
          "cache_control": {"type": "ephemeral"}
        },
        {
          "type": "text",
          "text": "问题：这份文档里的 X 是什么意思？"
        }
      ]
    }
  ]
}
```

两个 `cache_control` 标记了两个**断点**（最多 4 个）。服务会：

1. 算到第一个断点 → 把这一段的中间状态存下来
2. 算到第二个断点 → 把这一段的中间状态也存下来（追加，不覆盖前一段）
3. 对最后的"问题：..."正常做 prefill，生成回答

下次请求**前两段一字不差**，直接命中两段缓存，从最后的"问题"开始算。断点本身不收费——只按实际写入/读取的内容计费。2026 年起还有**自动模式**：在请求顶层放一个 `cache_control`，服务自动把断点打在最后一个可缓存块上、并随对话增长自动前移，官方现在推荐多轮对话首选它（Bedrock / Vertex 上暂不支持，仍需显式断点）。

**成本模型**：写入 1.25× 常规价、命中 0.1×——**同一前缀总共请求 ≥ 2 次（首次写入 + 之后命中 ≥ 1 次）就回本，越多越值**（第 1 次写入付 1.25×、第 2 次命中付 0.1×，合计 1.35× < 不缓存的 2×；以上按默认 5 分钟 TTL 算。1 小时 TTL 写入 2×，总共 ≥ 3 次回本：2 + 0.1×2 = 2.2× < 3×）。TTL 是"命中即续期"：持续有请求命中，笔记就一直不过期。

三个显式断点模型特有的机制，Agent 场景尤其要知道：

- **失效有级联顺序**：缓存键按 tools → system → messages 逐级构建。改工具定义 → 三级全灭；改 `tool_choice`、增删图片、开关 thinking → 只有 messages 段失效，tools/system 段的缓存还在
- **回看窗口有限**：读取时从断点向前最多回看 20 个 content block 找此前写过的条目。Agent 单回合塞进大量 tool_use/tool_result 块时会**静默错过命中**——长回合要在中部加断点
- **并发首请求会撞车**：缓存条目在第一个响应开始生成后才可读。同时到达的相同前缀请求都按未命中处理——需要并行扇出时，先等第一个请求开始返回，再发其余的

### 4.2 OpenAI：自动开，自动判

不用打标记。prompt ≥ 1024 token 时自动尝试缓存（工具定义、结构化输出 schema 都计入这个门槛），按前缀精确匹配；请求先按前缀哈希（约取前 256 token）路由到某台机器，再在那台机器上找缓存——所以它还提供 `prompt_cache_key` 参数让你主动把同前缀的请求聚到一起（§6.2）。

**成本模型**：写入 0 溢价；命中折扣随代际提升——gpt-4o 代省 50%、GPT-4.1 代省 75%、GPT-5 系省 90%（按代际的完整口径见[深入 03](03-模型与工具场景化最佳实践.md)）。命中量在响应里逐请求返回（`usage.prompt_tokens_details.cached_tokens`），是一等公民字段——你**能精确知道每次命中了多少**，不可控的是"没法指定缓存哪段、没法手动清除、没法预知会不会命中"。

保留时长有两档：默认的内存缓存不活跃 5–10 分钟后驱逐（off-peak 最长 1 小时）；2025 年底起有 `prompt_cache_retention: "24h"` **延长保留**——显存满了就把 KV 张量卸载到 GPU 本地存储，最长保留 24 小时，**同样零溢价**（GPT-4.1 与 GPT-5 全系支持；非零数据保留组织现已默认 24h）。

### 4.3 Google Gemini：隐式默认开 + 显式当资源

Gemini 是两套并存，官方措辞把差别说得很直白：

- **隐式缓存（implicit）**：Gemini 2.5 起默认开启，行为与 OpenAI 同型——自动按前缀匹配、命中折 90%、不收存储费，但官方明说 **"no cost saving guarantee"**（尽力而为，不保证）
- **显式缓存（explicit）**：把要缓存的内容先创建成 cached content 资源、拿到 ID，后续请求引用这个 ID。创建按标准输入价（无写入溢价）+ 按"缓存大小 × 存储时长"付**租金**（Flash 系 $1 / Pro 系 $4.5 每 MTok·小时），命中按 0.1× 计费。默认 TTL 1 小时，可任意加长；到点即删、不随命中自动续期，要延长得主动 update。官方对它的措辞是 **"cost saving guarantee"**——这是三家里唯一能写进成本 SLO 的缓存

**适合显式缓存的场景**："创建一次、查询很多次"（比如把一本大书做成 cache，让用户对它问一整周）。注意一个演进信号：Google 2026 年主推的新 Interactions API **只支持隐式缓存**，显式缓存留在被标记为 legacy 的旧 API 上——把架构押在显式缓存上的团队要盯紧这条线。

另外一条路线值得知道：国产厂商普遍走"全自动 + 两档价"——DeepSeek 直接把 cache-hit / cache-miss 做成两个单价，差 50×（快照数字见[深入 03](03-模型与工具场景化最佳实践.md)）。

### 4.4 对照表（2026-07 快照 🕒）

| 维度 | Anthropic | OpenAI | Google Gemini |
|---|---|---|---|
| 触发方式 | 显式 `cache_control` 断点（可选自动模式） | 自动（≥1024 token） | 隐式默认开 + 显式 cached content |
| 最小可缓存前缀 | 随模型 512–4096 token（趋势：新模型门槛更低；低于门槛静默不缓存、无报错） | 1024 token | 2.5 系 2048 / 3.x 系 4096 token |
| 写入成本 | 1.25×（5 分钟 TTL）/ 2×（1 小时 TTL） | 无溢价 | 隐式无；显式按标准输入价 + 存储租金 |
| 读取成本 | 0.1×（省 90%） | 按代际：~0.5× / ~0.25× / GPT-5 系 ~0.1× | 0.1×（2.5 及更新模型） |
| 保留时长 | 5 分钟（命中续期）/ 可选 1 小时 | 5–10 分钟滚动（off-peak 至 1h）；可选 24h（零溢价） | 隐式尽力而为；显式默认 1h、可自定 |
| 共享范围 | workspace 级（2026-02 起；Bedrock / Vertex 上仍为组织级） | 组织级 | 项目级 |

> [!NOTE]
> 表中全是会漂移的厂商参数，以各家官方文档为准——本表自己就是证据：Gemini 隐式缓存 2025-05 上线时折扣 75%、一年后变 90%；Anthropic 最小前缀门槛按模型多次调整。两个容易误判的点：① Anthropic 缓存按 workspace 隔离——同 workspace 内轮换 API key 不丢缓存，但拆分 workspace 会拆分缓存；② "共享范围"这一行不只是产品细节，它是安全边界（§6.6）。

## 5. 缓存什么时候作废：失效的七种形态

这是 SRE 最该关心的部分。**失效本身不会触发任何报警**——请求依然成功、依然返回正确答案，只是延迟变长、账单变贵。所以必须主动监控（§6.4）。

排障前先排除三种"根本没用上缓存"的情况：低于最小可缓存门槛（§4.4，静默不缓存）、并发首请求撞车（§4.1）、超出回看窗口（§4.1）。剩下的才是"写进去了、后来作废"，按下面七种形态排查。

### 5.1 TTL 过期（时间到了）

每段缓存有保质期。Anthropic 默认 5 分钟、**命中续期**——只要持续有请求命中，TTL 一直被刷新；一旦冷下来，到点失效。OpenAI 是不活跃 5–10 分钟后驱逐的滚动窗口。另注意：时间没到，也可能因为容量被提前挤掉——OpenAI 与自建侧尤甚（§5.7）。

**踩坑**：调用间隔比保留时长还长的低频场景，**永远命中不了**——开了 caching 等于白付写入溢价。

**对策按厂商分化了**（这也是 2026 年选型的新变量）：
- Anthropic：改用 1 小时 TTL（写入 2×、同一前缀总共请求 ≥ 3 次回本）；再低频就发 `max_tokens: 0` 的**官方预热请求**——只做 prefill 和缓存写入、不生成输出，比"心跳发个真请求"便宜得多。预热的钱值不值，照样要算
- OpenAI：`prompt_cache_retention: "24h"` 零溢价，10 分钟一次的低频调用天然覆盖——但保留时长不是 SLA
- Gemini：低频 + 大上下文是显式缓存的主场——租金换确定性
- 都不划算就干脆不开缓存

### 5.2 前缀变了（文本动了）

§3.2 已经讲过：前面任何一个 token 变化都会让后面全部失效。生产里最常见的几个犯罪现场：

- 系统提示里嵌了当前时间或当天日期
- 系统提示里嵌了用户 ID / 会话 ID
- 系统提示开头加了版本号 / A/B 标签
- 中间插了一段动态生成的"今日提示"
- 对话历史做滚动截断 / 摘要压缩 → messages 段前缀每轮都变，历史轮次永远 miss（缓存要求历史 **append-only**；"截断省 token"和"缓存省钱"是一对要显式权衡的账）

### 5.3 请求结构变了（文本没动，配置动了）

比 5.2 更隐蔽的一类：prompt 模板一个字没改，缓存照样失效——

- **工具定义改了**（新增工具、改描述、调参数 schema）→ 前缀从最开头变化，全量失效。**工具定义变更要纳入发布评审的"缓存影响分析"**，和改系统提示同等对待（评审动作的流程化见[深入 21](21-Prompt作为运维对象.md)）
- `tool_choice`、thinking 开关、增删图片 → Anthropic 上只废 messages 段（tools/system 段还在），但命中长度骤降的现象是一样的
- JSON 序列化 key 顺序不稳定（§3.4）→ 每次请求都是"新前缀"

### 5.4 路由变了（实例换了）

推理服务是分布式的，缓存通常**只存在于具体某台 GPU 实例的内存里**。下次请求被路由到另一台实例，prompt 完全一样也会 miss。

- **托管 API**：厂商尽量做前缀亲和路由（OpenAI 明确按前缀哈希路由，还给了 `prompt_cache_key`；超过约 15 请求/分钟的同键流量会**溢出**到其他机器，命中率下降——见 §6.2），但**没有 SLA 承诺**。流量突增、实例扩缩容时命中率会下降
- **自建推理服务**：必须部署**缓存感知路由**，否则缓存机制名存实亡。2026 年已有成熟开源组件（llm-d、SGLang Router、NVIDIA Dynamo、AIBrix——见 §6.2），不要自研简单的前缀 hash 亲和

### 5.5 实例重启 / OOM / 发版

部署新版、crash 恢复、扩容新起的实例 → 缓存为空，需要"预热期"才能恢复命中率。**部署窗口期成本和延迟都会上扬**——发版前一定要预期到。

### 5.6 模型版本切换

缓存是**模型绑定**的。换模型版本（甚至换量化方式）后，旧缓存全部作废。**灰度发布期间命中率会两端都下降**：旧版流量减少导致旧缓存冷却失效，新版还没攒起新缓存。

### 5.7 容量驱逐（时间没到，位置没了）

TTL 不全是承诺：Anthropic 的口径是"**至少** 5 分钟、命中续期"（下限）；OpenAI 的"不活跃 5–10 分钟"本来就是尽力而为，高峰期会更早驱逐；自建侧 vLLM / SGLang 的前缀缓存**根本没有 TTL**，是纯 LRU：高并发峰值把冷前缀块挤出显存，"命中率随负载升高而下降"的经典现象就是这一条。缓存留多大显存、和并发怎么分账，见 §6.3。

## 6. 对 SRE 的工程含义

### 6.1 必须监控的指标

| 指标 | 说明 | 警戒线 |
|---|---|---|
| Cache hit rate | `cache_read / (cache_read + input + cache_creation)`（Anthropic 口径，换算见表下） | 较基线相对下降 20%+ 且持续（如 15 分钟）即告警，绝对阈值按业务自定 |
| Cache hit prefix length | 平均命中多少 token | 突然变短 → 前缀设计被某个改动破坏了 |
| Cache write rate | 每秒写入次数 | 突增 → 前缀在变（有人加了动态字段） |
| Per-session / per-channel hit rate | 单会话 / 单上游通道 | 某些维度命中率异常低，针对性排查 |
| TTFT | 首包延迟（见[深入 01](01-首包延迟与吞吐的影响因素.md)） | **缓存失效最早的信号**：TTFT 突升先于账单异常数小时到数天 |
| Cost savings | 按 token 折算的节省 | 业务层追踪 |

三个最常见的对账误读：

- Anthropic 的 `input_tokens` **不是总输入**——它只统计最后一个断点之后、没参与缓存的残段。总输入 = `input_tokens + cache_creation_input_tokens + cache_read_input_tokens`，hit rate 的分母必须用三者之和
- 用了 1 小时 TTL 后，写入还有细分（`usage.cache_creation.ephemeral_5m_input_tokens / ephemeral_1h_input_tokens`），两者之和恒等于 `cache_creation_input_tokens`——可以拿这个恒等式做计费日志的自校验
- 上面的三字段口径是 Anthropic 的。OpenAI / Gemini 的 cached 字段（`cached_tokens` / `cachedContentTokenCount`）是 prompt 总量的**子集**——它们的 hit rate = cached ÷ prompt 总量，别把 cached 再加进分母，加了就是重复计数

### 6.2 命中率工程：把命中当成一个可设计的量

**托管 API 侧**，你能动的旋钮比想象的多：

- **断点位置**（Anthropic）：多轮对话让断点随轮次前移（或直接用自动模式）；Agent 长回合在中部加断点，避开 20-block 回看窗口
- **请求分桶**（OpenAI）：按"业务线 / 租户 / prompt 模板"设计 `prompt_cache_key` 粒度，让每个（前缀, key）组合的流量保持在约 15 请求/分钟以下——粒度太粗会溢出、太细会分散
- **时间聚簇**：相似前缀的请求尽量在时间上凑在一起发（批处理排序按前缀分组——离线场景把命中率从 10% 提到 85% 的做法见[深入 13](13-离线批量推理.md)）
- **并发扇出前先"探路"**：等第一个请求开始返回、缓存条目可读之后，再放剩余并发（§4.1 的撞车问题）

**自建推理侧**，vLLM（block 级 hash 表，V1 引擎默认开启）和 SGLang（RadixAttention，token 级 radix tree）都自带前缀缓存，真正的工程量在**跨实例路由**：路由器维护各实例缓存内容的近似索引，把请求打给"最可能命中的那台"，同时拿负载均衡兜底——llm-d（GKE Inference Gateway 采用）、SGLang Router、NVIDIA Dynamo、AIBrix 都已可生产使用。代价是负载可能出现热点：**命中率和均衡性之间的权衡，这些组件把它参数化了，但没有替你做掉**。

### 6.3 容量规划：缓存和并发是此消彼长的

缓存**占显存**。一个 70B 模型、128k context 的完整 KV cache 大约 **40 GB**（口径见[深入 05](05-LLM推理服务的容量规划.md)）。

> **给缓存留多少显存 ↔ 能服务多少并发——此消彼长。**

留得多：命中率上升、单次请求便宜、首包延迟低；但能跑的并发数下降。这是 AI 时代容量规划的新维度。两个 2026 年的新变量：

- **分层存储正在松动这个约束**：把 KV cache 下沉到 CPU 内存 / SSD / 远端存储池（LMCache、Mooncake、NVIDIA Dynamo 的四级层次）。Kimi 的 Mooncake 把这件事概括成一句话——"用存储换计算"；vLLM + Mooncake 在 Agent 类负载下给出过 TTFT 降一个数量级以上的公开数字
- **限流账各家方向相反**：Anthropic 官方把"缓存命中不占用速率限制"列为 1 小时 TTL 的适用理由之一；OpenAI 则明确命中 token **照常计入 TPM**。同一套流量换通道，容量余量完全不同——网关容量规划要分家算

### 6.4 失效的静默性

再强调一次：缓存失效**没有报警**——请求不报错，延迟变长但不过 timeout 阈值，传统的错误率 / 超时报警全程静默。你真正要记的是三条检测通道的时间线差：

- **TTFT 突升**：分钟级、逐请求可见，最早的前哨（§6.1）
- **Hit rate 仪表盘**：随聚合窗口滞后小时级
- **成本 / 账单**：财务数据滞后天级

这就是为什么 §6.1 要给 hit rate 单设警戒线——等账单说话，事故已经跑了几天。

### 6.5 多上游网关位的特殊风险

> **一句话**：网关后面挂着三家上游，缓存策略不能"一刀切"。同一段 prompt 走 Anthropic 和走 OpenAI，缓存账单可能相差数十个百分点——差多少取决于对端代际、重用率和保留时长。网关不知道是哪类工作负载就做自动路由，等于在烧钱。

如果你做的是**通用 LLM 网关**（一个 endpoint 后面挂着 OpenAI / Anthropic / Gemini / 自建模型若干家），三家差异就从"接口细节"升级成"生产风险"。

**风险一：写入溢价与保留时长都不对称**

把同一个用户从 OpenAI 通道切到 Anthropic 通道，看似只换了 model 名字，账单完全不同：

- OpenAI 通道：写入 0 溢价，命中折扣随代际提升（gpt-4o 代省 50%、GPT-4.1 代省 75%、GPT-5 系省 90%），还可以零成本把保留拉到 24h
- Anthropic 通道：写入溢价 25%（1h TTL 是 100%），命中省 90%

结论要按对端代际和调用节奏分开算。对端是 gpt-4o / GPT-4.1 代时，**高重用场景**（多轮对话、RAG 反复命中）切到 Anthropic 反而便宜；对端是 GPT-5 系时读取折扣打平（都省 90%），Anthropic 恒多付写入溢价，账单只会渐近持平。而**低重用、低频场景**不管对端是哪代，切到 Anthropic 都直接涨价 25%——且 OpenAI 的 24h 保留让"低频"的判定线从 10 分钟放宽到了小时级，这一格的差距比 2025 年更大了。网关如果不区分工作负载与对端代际就做"自动路由"，账单会无缘无故上下浮动。

**风险二：归一化是个伪命题**

"在网关层抹平 caching 差异"听起来漂亮，做不到：

- Anthropic 要客户端显式打 `cache_control`，OpenAI 不需要 → 网关无法用同一种入参表达"我希望缓存这一段"
- OpenAI / Gemini 隐式路径是自动判定的，网关只能在响应里读到命中量，无法主动控制
- Gemini 的显式缓存是**独立资源**（先 create 拿 ID，再引用），跟前两家的"内联标记 / 全自动"模型根本不同，还有独立的存储计费

**结论**：网关层**只能做透传 + 观测，不要做归一化**。

> [!WARNING] 何时这条不再适用
> "不归一化"的前提是三家缓存语义互不兼容。当以下任一事件发生时重审本节：主流上游就缓存控制收敛出可互换的标准（例如 OpenAI 兼容接口原生接受 `cache_control` 语义并保证等效计费）；或你的网关收敛到单一上游家族。届时"网关层统一缓存参数"会从伪命题变成普通需求。

**风险三：计费回放与对账**

每家上游回包里"缓存命中量"的字段名与嵌套层级都不同：
- Anthropic：`usage.cache_read_input_tokens / cache_creation_input_tokens`（注意 `input_tokens` 只是断点后残段，§6.1）
- OpenAI：Chat Completions 是 `usage.prompt_tokens_details.cached_tokens`，Responses API 是 `usage.input_tokens_details.cached_tokens`——**同一家就有两套**
- Gemini：`usageMetadata.cachedContentTokenCount`（隐式与显式命中都走这个字段）

网关如果用一套统一 `usage` 结构落 DB，**默认会丢一两个字段**，月底对账对不上（字段并集原则见[深入 17](17-LLM网关的SRE视角.md)）。

**网关侧最小自检**：

- [ ] 计费日志同时记录 `prompt_tokens / cache_read_tokens / cache_write_tokens / completion_tokens` 四列。任一上游缺字段就填 0，**字段必须存在**；落库前先归一口径——`prompt_tokens` 一律记"含缓存在内的总输入"（Anthropic 通道 = 三字段之和，OpenAI / Gemini 的 cached 本来就是总量的子集）
- [ ] 同一会话 / 同一前缀族黏住同一个上游凭证（workspace / 组织 / 项目）与区域——**跨 workspace / 组织 / 项目**的多 key 轮询会把缓存打散成 N 份（同 workspace 内轮换 key 不丢缓存，§4.4），failover 重试按全量 miss 计入成本预算
- [ ] 网关对可缓存段做字节级透传，不要解析后重新序列化请求体（`cache_control` 块原样透传）——网关自己就是 §3.4 里"key 顺序不稳定"的最常见肇事者
- [ ] 不同 channel 的 hit rate 分开看，混在一起没意义
- [ ] 跨 channel 路由策略上线前，**先用真实工作负载回放算两边账单**（折扣、写入溢价、保留时长三个维度都要算），再决定是否切流
- [ ] 跟客户沟通时讲清"换通道 = 换缓存计费模型"，别让调用方以为"模型名一样就一样"

### 6.6 多租户下的安全：缓存是一条计时侧信道

缓存命中的请求 TTFT 显著更低——这个延迟差本身就是信息。斯坦福团队 2025 年发表的审计（《Auditing Prompt Caching in Language Model APIs》，ICML 2025）用统计假设检验测了 17 家 LLM API：8 家检测到缓存，其中 **7 家当时跨用户全局共享**——攻击者原则上可以通过响应时间探测"有没有别人发过以某段文字开头的 prompt"（名单里的 OpenAI 端点是 embeddings API，不是 chat；同期 Anthropic / OpenAI 的对话模型检测到的是按用户/组织级隔离，未跨用户全局共享）。负责任披露后，多家厂商收紧为按组织/workspace 隔离——**§4.4 表里"共享范围"那一行，就是这项研究留下的安全边界**。

对自建和网关位的含义更直接：

- 自建 vLLM / SGLang 的前缀缓存默认**在实例内全局共享**——多租户平台等于复刻了当年被审计出的漏洞形态：租户 A 可以用计时探测租户 B 的 prompt 前缀
- 缓解手段是现成的：vLLM 支持请求级 `cache_salt`（按租户注入隔离盐）、hash 算法可切 sha256 防碰撞；更彻底的是按租户拆缓存命名空间或拆实例
- 代价同样明确：**隔离降低命中率**。这是一笔"安全 vs 成本"的显式账，应该由平台按租户数据敏感度来定，而不是默认全共享
- 托管 API 也别掉以轻心：组织 / workspace **内部**同样是共享的——建在其上的多租户 SaaS，终端租户之间的侧信道由你负责，厂商没有租户级隔离旋钮。缓解只能靠别把租户敏感内容放进公共可缓存前缀，或按敏感度拆 workspace（代价同样是命中率）

## 7. 实战：哪些场景最受益

**高受益场景**——前缀长、重复用、变化只在尾巴：
- **Agent / 多轮对话**：每一步都要重放全部历史（系统提示 + 工具定义 + 前几轮），没有缓存的 Agent 在成本上几乎不可运营（[深入 04](04-为什么简单你好也消耗数万token.md) 算过这笔账）；断点随轮次往后推进，历史轮次全部命中——前提是历史 **append-only**，一旦做滚动截断 / 摘要压缩就不成立（§5.2）
- **RAG**：固定文档放前面，变化查询放后面（§1 的场景）
- **长系统提示的对话产品**：Few-shot 示例、人设定义放前面
- **代码助手**：仓库/文件上下文放前面

**低收益甚至负收益**：
- 一次性短 prompt（低于最小可缓存门槛的连报错都没有，静默不缓存）
- 每次前面都在变的场景（嵌了时间、随机值、用户 ID）
- 调用频率比保留时长还低的场景——但注意这条的边界在漂移：OpenAI 24h 保留、Gemini 显式缓存都在把"低频"的门槛往宽里推（§5.1）

## 8. 常见陷阱 Checklist

- [ ] 不要在 prompt 最前面放时间戳 / UUID / 随机值
- [ ] 不要每次改版本号放在系统提示开头
- [ ] 用户 ID / session ID 没必要就别放前面（放前面 = 缓存按用户隔离）
- [ ] **工具定义变更要过"缓存影响分析"**——改一行工具描述 = 全量失效，和改系统提示同等对待
- [ ] **要监控实际命中率，不要假设设了 `cache_control` 就一定命中**（分母用三字段之和，§6.1）
- [ ] 对账时别把 Anthropic 的 `input_tokens` 当总输入——它只是断点后的残段
- [ ] 检查 SDK 的 JSON 序列化 key 顺序是否稳定（Swift / Go 易踩）
- [ ] 模型灰度发布时预期到缓存失效的成本冲击
- [ ] 自建服务时部署缓存感知路由（用成熟组件，别自研 hash 亲和），否则缓存机制名存实亡
- [ ] 多租户平台不要默认共享缓存命名空间——计时侧信道是真实攻击面（§6.6）
- [ ] 不要以为缓存的是"答案"——`temperature > 0` 时同 prompt 答案也不同，缓存的是中间状态（KV cache）
- [ ] 写入比读取贵——冷门 prompt 不要标记缓存

## 9. 想再往下挖的术语索引

如果你想把直觉补成完整的工程心智模型，下面这些术语都是入口（这一节为了平铺直觉，没有展开）：

- **KV cache / Prefill / Decode**：模型推理的两个阶段。详见[深入 01 · 首包延迟与吞吐](01-首包延迟与吞吐的影响因素.md)；"为什么历史 K/V 不变"的物理原理见[科学 01](../科学/01-Attention与Transformer的SRE视角.md)
- **PagedAttention**（vLLM 论文）：把 KV cache 按页管理，让多请求高效共享内存
- **RadixAttention**（SGLang）：用 radix tree 在 token 级管理多请求间的共享前缀——与 vLLM 的 block 级 hash 是两种粒度路线（框架对比见[深入 19](19-模型服务框架对比.md)）
- **Prefix Caching**：开源推理服务（vLLM / SGLang / TGI）实现 prompt caching 的内部叫法
- **前缀感知路由**（prefix-aware / KV-cache-aware routing）：llm-d、SGLang Router、NVIDIA Dynamo、AIBrix——把"请求发给最可能命中的实例"做成路由层组件
- **KV cache 分层与分离式存储**：LMCache、Mooncake（USENIX FAST'25 Best Paper，"用存储换计算"）、NVIDIA Dynamo KVBM 的 GPU→CPU→SSD→远端四级层次
- **cache_salt / 缓存隔离**：vLLM 的多租户缓存隔离机制（§6.6 的缓解手段）
