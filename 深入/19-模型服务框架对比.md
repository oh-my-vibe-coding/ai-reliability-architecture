# 深入 19 · 模型服务框架对比：引擎与编排的两层选型

> [← 返回目录](../README.md)  ·  相关：[深入 01 · TTFT 与吞吐](01-首包延迟与吞吐的影响因素.md)  ·  [深入 02 · Prompt Caching](02-Prompt-Caching原理.md)  ·  [深入 05 · 容量规划](05-LLM推理服务的容量规划.md)  ·  [深入 20 · 分布式推理](20-单卡装不下的大模型分布式推理.md)

> [!NOTE]
> 2026 年自建推理的选型是**两层**：先选**引擎**（vLLM / SGLang / TensorRT-LLM——单实例把 token 算得快），上了规模再选**编排层**（llm-d / NVIDIA Dynamo / AIBrix——多实例把请求送到缓存和拓扑最优的那台）。这一章从 SRE 关心的维度集中对比两层，回答"到底该选哪个、什么时候需要第二层"。逐版本的特性快照以[深入 03 §4.5](03-模型与工具场景化最佳实践.md) 为准，本章给选型框架。

---

## 0. 一句话定位（2026-07 快照 🕒）

| 框架 | 一句话 | 典型用户 |
|---|---|---|
| **vLLM** | 开源社区的事实标准，PagedAttention 发明者；硬件插件生态最广 | 绝大多数自建推理场景（默认起点） |
| **SGLang** | RadixAttention（token 级前缀共享）+ 大规模 EP（专家并行）/ PD（prefill-decode 分离）部署的工程标杆（原理见[深入 20](20-单卡装不下的大模型分布式推理.md)） | Agent / 深共享前缀负载、DeepSeek 类 MoE 大规模部署 |
| **TensorRT-LLM** | NVIDIA 官方引擎，2025 起 PyTorch 原生、无编译门槛 | 全 NVIDIA 栈 + 深度调优诉求 |

两条退场与例外的注记：

- **TGI 已退场**：HuggingFace 官方于 2025-12 把 TGI 转入维护模式（只收 bugfix），README 明确推荐迁移到 vLLM / SGLang。它不再是选项，只是迁移对象（§6.3、§7）。
- **边缘 / 本地**（Apple Silicon、笔记本）不在本章两层框架内——llama.cpp 系是事实标准，见[深入 03 §4.5](03-模型与工具场景化最佳实践.md)。

## 1. 第一层：引擎对比

### 1.1 性能（2026-07 快照 🕒）

| 维度 | vLLM | SGLang | TensorRT-LLM |
|---|---|---|---|
| 吞吐 / TTFT | 同一梯队 | 同一梯队 | 同一梯队 |
| 前缀缓存 | block 级 hash（默认开启，未命中近零开销） | RadixAttention：token 级 radix tree（默认开启） | paged KV + block reuse |
| 深共享前缀负载（Agent 多轮） | 按整 block 命中 | **任意长度前缀命中**，树状多分支共享 | block reuse |
| 投机解码 | EAGLE-3 / MTP（多 token 预测）/ n-gram | EAGLE-3 / MTP / n-gram / 独立 draft | EAGLE / MTP / n-gram |
| 大规模 EP + PD 分离 | Wide-EP 已有官方实践 | **工程标杆**（DeepSeek 类部署的首选） | Disaggregated Serving（Beta） |

> [!IMPORTANT]
> "谁吞吐最高"这个问题在 2026 年已经**没有稳定答案**。早年"SGLang 比 vLLM 快 2-4×"的数字出自 2024 年论文，对比的是 V0 时代、前缀缓存默认关闭的旧 vLLM；vLLM V1（2025-03 起默认）把前缀缓存做成默认开启、未命中近零开销后，中立的持续基准（InferenceMAX 夜跑）显示三家引擎处于**同一梯队**，差距随负载形态在百分之十几到三十之间摆动、随版本互有胜负。真正稳定的差异不在吞吐数字，而在下面三条结构性维度：前缀缓存的**粒度路线**（token 级 radix tree vs block 级 hash——机制对比见[深入 02 · §6.2](02-Prompt-Caching原理.md)）、**大规模 EP/PD 的工程成熟度**、**硬件与生态绑定**。

### 1.2 功能（2026-07 快照 🕒）

| 维度 | vLLM | SGLang | TensorRT-LLM |
|---|---|---|---|
| Continuous batching + chunked prefill | ✅（V1 内建，默认） | ✅ | ✅（in-flight batching） |
| Structured output | ✅ guided decoding | ✅ 原生 JSON/Regex 约束 | ✅ guided decoding |
| LoRA 多 adapter | ✅ 热加载 | ✅ 正式功能（动态 load/unload、adapter 固定、`model:adapter` 语法） | ✅（HF / NeMo 格式） |
| PD 解耦 | ✅ KV connector（NIXL / LMCache / Mooncake）+ 外部编排 | ✅（Mooncake / NIXL 传输） | Beta（配合 Dynamo） |
| 量化 | AWQ / GPTQ / FP8 | AWQ / GPTQ / FP8 | FP8 / FP4 / INT4 原生最优 |
| OpenAI 兼容 API | ✅ 原生 | ✅ 原生 | ✅（`trtllm-serve`） |

### 1.3 运维（2026-07 快照 🕒）

| 维度 | vLLM | SGLang | TensorRT-LLM |
|---|---|---|---|
| 部署复杂度 | 低（pip install + 一行启动） | 低 | 中（PyTorch 后端，HF checkpoint 直接 `trtllm-serve` 起服务；**已无编译步骤**） |
| 模型支持广度 | 极广（社区驱动，新架构一周内有 PR） | 广（常用架构全覆盖，day-0 支持热门模型） | 较广（宣称热门模型 day-0，长尾靠 NVIDIA 节奏） |
| **硬件多样性** | **最广**：ROCm 一等平台；昇腾 / Gaudi / TPU 官方插件 | NVIDIA + AMD 深度合作（MI300X/MI355X day-0） | NVIDIA-only |
| 发版节奏 | 约 2 周一个 minor | 约 2-6 周 | 周级 rc + 季度级正式版 |
| K8s 部署 | production-stack / llm-d 等官方生态项目 | SGLang Model Gateway + K8s 发现 | 配合 Dynamo / Triton |
| 可观测性 | Prometheus metrics + 日志 | Prometheus + OTel | trtllm-serve 自带指标端点（迭代统计需 `enable_iter_perf_stats` 显式开启，Prometheus 口径较简）；Triton 指标族仅在配合 Triton/Dynamo 部署时可用 |

三个 2026 年的关键更正（如果你对这三家的印象来自 2024-2025 年初的资料）：

- **TensorRT-LLM 不再是"编译式框架"**。1.0（2025-09）起 PyTorch 原生后端成为稳定默认；到当前主干（1.3 时代），TensorRT 引擎编译路径（`trtllm-build`）已完全移除，PyTorch 是唯一执行后端，HF checkpoint 直接加载起服务。"预留 1-2 周编译上线周期"的旧印象作废——它的真实门槛变成了**调优复杂度和 NVIDIA-only 锁定**。（顺带纠正一个更老的讹传：ONNX 导出从来不是 TRT-LLM 的模型接入路径。）
- **SGLang 的 LoRA 已是正式功能**，不再是"实验性"：动态加载卸载端点、adapter GPU 固定、OpenAI 兼容的 `model:adapter` 调用语法俱全。
- **"每轮重算 prefix"对 2026 年的 vLLM 是错的**：V1 引擎下前缀缓存默认开启且未命中近零开销。

### 1.4 容错与稳定性

| 维度 | vLLM | SGLang | TensorRT-LLM |
|---|---|---|---|
| KV 不足时 | **请求级抢占（preemption）**：驱逐请求、默认重算方式重新调度 | RadixAttention 的 LRU 驱逐 + 抢占 | paged KV 动态管理 |
| 关键监控信号 | `vllm:num_preemptions_total` 突增 = 容量不足 | cache 驱逐率 / 队列深度 | trtllm-serve 迭代统计（KV 占用 / in-flight batching 深度）；配 Triton/Dynamo 时才有 Triton 指标族 |
| 请求超时 / 优雅关闭 | ✅ | ✅ | ✅ |
| 故障恢复 | 外部（K8s 重启） | 外部 | 外部 |

> [!WARNING]
> 老资料常说"vLLM OOM 时自动降级（减少 max_num_seqs）"——**没有这个机制**。`max_num_seqs` 是静态配置；真实机制是抢占：KV 显存不足时驱逐在途请求稍后重算，代价是被抢占请求的延迟大幅上升。抢占频繁 = 容量信号，处理手段是调 `gpu_memory_utilization` / `max_num_batched_tokens` 或加卡（容量账见[深入 05](05-LLM推理服务的容量规划.md)），不是等引擎"自动降级"。

## 2. 第二层：编排层（多实例之后的选型）

单实例引擎解决"token 算得快"；多副本之后的问题变成"**请求送到哪台**"——送错实例，前缀缓存全部 miss（机制见[深入 02 · §5.4/§6.2](02-Prompt-Caching原理.md)）。2026 年这一层已经有了成建制的开源选项，**不要自研**。

### 2.1 什么时候需要编排层

单副本、单机多卡：**不需要**。普通 K8s Deployment + HPA 足够，编排层是负资产。出现以下任一信号再上：

1. **多副本 + 长共享前缀负载**（多轮对话 / Agent / 带大系统提示的 RAG）——KV 感知路由才有收益，且收益强依赖前缀命中率
2. 要做 **PD 分离**（原理与算例见[深入 20 §4-§5](20-单卡装不下的大模型分布式推理.md)）
3. 模型大到**多机**（MoE / 超长上下文）
4. **异构 GPU 池**要做成本调度
5. **高密度多模型 / 多 LoRA** 服务

信号 1 是五个里唯一需要量化判断的：先看随机路由下单实例的前缀缓存命中率（引擎自带指标，口径见[深入 02 · §6.1](02-Prompt-Caching原理.md)）——若随机路由下命中率本来就不低，说明单实例缓存已经够用；命中率低但负载确有长共享前缀，才是 KV 感知路由的收益空间。拿不准就用 §4 的 inference-perf 做一次带编排层 / 不带编排层的整栈对照压测，用数字回答"上不上"。

### 2.2 编排层选项（2026-07 快照 🕒）

| 项目 | 归属与治理 | 一句话定位 |
|---|---|---|
| **llm-d** | Red Hat + Google + IBM 发起，2026-03 进 CNCF Sandbox | K8s 原生、中立治理；GKE Inference Gateway 与 Red Hat AI 两条托管路径；上游仍 0.x |
| **NVIDIA Dynamo** | NVIDIA，1.0（2026-03）官方宣布 production-ready | 引擎无关（vLLM / SGLang / TRT-LLM 三后端）；可跑在 K8s 之外；KVBM 四级 KV 分层（GPU→CPU→SSD→远端）是独有卖点 |
| **AIBrix** | 字节开源，vllm-project 官方子项目 | 高密度 LoRA 交付 + 异构 GPU 成本优化；已支持 vLLM / SGLang 多引擎 |
| **SGLang Model Gateway** | SGLang 官方（前身 Router） | Rust 实现：cache-aware 负载均衡、PD 路由、熔断限流、mTLS |
| **vLLM production-stack** | vllm-project 官方 | 轻量 K8s Helm 起步路径 |
| **Ray Serve LLM / KServe** | Ray / KServe 社区 | 已有平台的搭车路径：Ray 的前缀感知路由仍 alpha；KServe 的 LLMInferenceService **内嵌 llm-d** |

其中 KServe 内嵌 llm-d 这件事值得单独一提——它说明编排层是"组合件"，不是又一个竞品。

> [!TIP]
> **押注标准接口，保留退路**：K8s Gateway API Inference Extension（`InferencePool`，v1.0 于 2025 年下半年 GA）是 GKE、Envoy、Istio、NGINX、Higress 等各家网关都在实现的收敛点。选实现了这个接口的路由方案，等于保留了"换编排层不换网关"的退路——这是 SRE 视角最值钱的一条选型判据。

编排层内部怎么权衡命中率与均衡性，属于缓存命中率工程，见[深入 02 · §6.2](02-Prompt-Caching原理.md)；本节只回答"选哪家"。

## 3. SRE 选型决策树

```text
要自建推理服务
  │
  ├─ 第一层：选引擎（单实例）
  │   ├─ 默认 → vLLM（社区事实标准；非 NVIDIA 卡/国产化场景插件生态最广）
  │   ├─ Agent / 深共享前缀负载，或对标 DeepSeek 类大规模 EP+PD
  │   │    → 评估 SGLang（用自己的 workload 复测，见 §4）
  │   ├─ 全 NVIDIA 栈 + 深度调优诉求 → TensorRT-LLM（无编译门槛，换 NVIDIA-only）
  │   └─ 还在跑 TGI → 不是选型，是迁移（§6.3）
  │
  └─ 第二层：要不要编排层（多副本之后）
      ├─ 没触发 §2.1 的五个信号 → 不要，K8s 原生够用
      ├─ K8s 原生 + 中立治理 → llm-d（上游仍 0.x：直接跑上游=承担早期风险，生产优先经 GKE Inference Gateway / Red Hat AI 托管形态）
      ├─ 全 NVIDIA 栈 / 不在 K8s 上 / 要 KV 四级分层 → NVIDIA Dynamo
      ├─ vLLM 生态 + 高密度 LoRA / 异构 GPU 成本 → AIBrix
      ├─ 引擎选了 SGLang → SGLang Model Gateway（官方同生态，cache-aware LB / PD 路由开箱即用）；要引擎无关再看 Dynamo / AIBrix（两者均支持 SGLang）
      ├─ 已有 Ray / KServe 平台 → 搭车（Ray 路由仍 alpha；KServe 内嵌 llm-d）
      └─ 无论选谁 → 优先支持 Gateway API Inference Extension 的方案
```

**默认推荐**：从 vLLM 起步，文档最全、坑最少。遇到明确瓶颈（深前缀负载命中率、超大规模 EP/PD）再评估 SGLang；追求极致且接受 NVIDIA 锁定再看 TRT-LLM；上编排层看 §2.1 的**触发信号**，不看技术新鲜度。

> [!WARNING] 何时这条不再适用
> - 某引擎在**你的 workload 复测**中稳定领先 30%+ 且维护节奏健康 → 重估"默认 vLLM"；
> - Gateway API Inference Extension 停止演进或被别的标准取代 → 重审"押注标准接口"；
> - 引擎原生长出成熟的集群模式（内置跨实例路由与 PD 编排）→ 两层选型合并回一层，本章框架需要重画。

## 4. Benchmark 方法论：别信公开数字，自己测

公开对比数字的半衰期只有几个月（本章 2024 年版的"2-4×"就是教训）。2026 年的正确姿势是三层工具对三层问题：

| 你要回答的问题 | 工具 | 说明 |
|---|---|---|
| 引擎参数怎么调 | `vllm bench`（serve / latency / throughput / sweep） | 引擎内置 CLI（此行以 vLLM 为例；SGLang / TRT-LLM 用各自内置 bench，或直接用 AIPerf 打端点），升级前后回归也用它 |
| 端点 SLO 扛不扛得住 | **AIPerf**（genai-perf 的继任者） | 客户端压测 TTFT / ITL / 吞吐，可打任意 OpenAI 兼容端点 |
| 编排层到底赚不赚 | **inference-perf**（K8s WG-Serving 出品） | 整栈多副本压测；编排层的收益只有**整栈压测**才测得出来——用单点工具会得出"编排层没用"的错误结论 |

要横比引擎，引用 **InferenceMAX**（SemiAnalysis，2025-10 起持续夜跑、开源可复现）这类中立持续基准，而不是任何一家的官方博客。测试负载必须还原你自己的前缀共享结构——前缀命中率是三家差距的最大变量（[深入 02 · §6.1](02-Prompt-Caching原理.md)）。

## 5. 安全与多租户：推理引擎的攻击面

这一节在 2024 年的选型文档里不存在，2025 年那一波 CVE 把它变成了必修课（2026-07 快照 🕒）：

- **CVE-2025-32444**（CVSS 10.0）：vLLM 的 Mooncake 集成经**未鉴权 ZeroMQ socket** 传输 pickle 数据导致远程代码执行（0.8.5 修复）
- **CVE-2025-47277**：PyNcclPipe 同类问题（内部通信面 RCE；0.8.5 修复）
- **CVE-2025-25183**（低危但有启发性）：prefix cache 用 Python 内建 hash，可构造 prompt 触发**缓存碰撞污染后续响应**——"前缀缓存是核心能力"的安全反面（0.7.2 修复）
- 此后 vLLM 累计发布 40+ 安全公告，Critical 级持续出现

给 SRE 的三条结论：

1. **推理引擎的内部通信面默认不设防**。vLLM 官方安全文档对 ZMQ、NCCL、KV transfer 这些内部通信面明说 "insecure by default"——不加密、不鉴权。官方安全模型就是**网络隔离**：跨实例通信必须锁在隔离网段，不能裸奔在共享内网
2. **OpenAI 兼容端点默认无鉴权**，`--api-key` 只是可选的单一静态 key。对外必须过网关（[深入 17](17-LLM网关的SRE视角.md)），多租户前缀缓存的隔离（`cache_salt`、计时侧信道）见[深入 02 · §6.6](02-Prompt-Caching原理.md)
3. 双周级发版节奏 × 持续 CVE = **版本管理是运维流程**，不是"有空再升"。固定版本（且不低于已知 CVE 修复线）+ 定期安全升级窗口 + 升级前后用引擎内置 bench（vLLM 为 `vllm bench`）做性能回归（引擎升级引入性能回归的事故并不罕见）

## 6. SRE 的迁移检查清单

无论迁往哪个引擎，标配三件事——(1) 灰度切流 + 新旧双跑对比（延迟分布 / 错误率 / 输出抽检），决策树见[练习 Unit3 · Week4 灰度回滚决策树](../练习/Unit3-推理SLO与静默降级/Week4-灰度回滚决策树.md)；(2) 回滚预案：旧引擎保留可秒级切回的部署，确认满足回滚时限要求；(3) 指标与告警口径迁移：旧引擎专属指标名在新引擎上不存在，逐条映射而不是照搬 dashboard。

### 6.1 从 vLLM → SGLang

- [ ] 确认你的模型架构在 SGLang 支持列表中
- [ ] 用你自己的 workload profile 复测吞吐（§4——不要信公开 benchmark，包括本章的定性结论）
- [ ] 监控前缀缓存命中率（决定实际收益，口径见[深入 02 · §6.1](02-Prompt-Caching原理.md)）
- [ ] 迁移 `vllm:*` 告警到 SGLang 对应指标：§1.4 的抢占容量告警（`vllm:num_preemptions_total`）需映射为 cache 驱逐率 / 队列深度
- [ ] 检查 structured output 行为是否一致（JSON schema 约束在框架间有微小差异）
- [ ] LoRA 场景：迁移到 SGLang 的动态 load/unload 端点与 `model:adapter` 语法

### 6.2 从任何框架 → TensorRT-LLM

- [ ] 确认 PyTorch 后端支持你的模型（HF checkpoint 直接加载，无编译步骤）
- [ ] 做数值一致性回归测试（kernel 差异可能引入输出漂移，参照[第 5 章 · 静默降级](../知识/05-AI推理服务的可靠性工程.md)）
- [ ] 评估量化版本的精度损失（FP8/FP4 实现与其他框架有差异）
- [ ] 确认接受 NVIDIA-only：换卡（AMD / 国产）= 换引擎

### 6.3 从 TGI → vLLM / SGLang（存量迁移）

- [ ] TGI 2025-12 起维护模式：不加新功能、不保证跟进新模型——制定**迁移时间表**，别等它出安全问题
- [ ] Medusa / n-gram 投机解码配置需重新评估（三个活跃框架的主流是 EAGLE-3 / MTP）
- [ ] 对齐 metrics 名称与 dashboards（TGI 与 vLLM 的指标语义不同）

## 7. 框架生命周期风险：TGI 的三年

TGI 从 2022 年底诞生、2023 年成为事实标准，到 2025-12 转入维护模式，只用了三年。它不是被"更快的框架"击败的——HuggingFace 的战略转向了"transformers 作为模型定义层、由 vLLM / SGLang 消费"，TGI 失去了在这套新战略里的位置。

选型要评估的不只是今天的 benchmark，还有项目背后组织的**战略利益是否与项目存续绑定**：vLLM（Linux 基金会旗下 PyTorch 生态）与 llm-d（CNCF）的中立治理、SGLang 背后的产业采用（超大规模 GPU 部署）、TRT-LLM 与 NVIDIA 卖硬件的绑定，都是比"当前吞吐"更长寿的判据。这也是[第 12 章 · 架构师的决策框架](../架构/03-架构师的决策框架.md)"可逆性"原则在框架选型上的应用。

## 8. 给 SRE 的一句话总结

> [!IMPORTANT]
> 前缀缓存已经**不是竞争点**，而是三家引擎的共同默认能力——差异只剩粒度路线（token 级 radix tree vs block 级 hash，见[深入 02](02-Prompt-Caching原理.md)）。真正的分水岭有三条：**大规模 EP/PD 的工程成熟度**（原理见[深入 20 §4-§5](20-单卡装不下的大模型分布式推理.md)）、**硬件路线宽度**、以及**要不要第二层（编排层）**。
>
> 默认从 vLLM 起步；深前缀 / 超大规模负载评估 SGLang；全 NVIDIA 栈再看 TRT-LLM。多副本之后按 §2.1 的五个信号决定是否上编排层，并优先选支持 Gateway API Inference Extension 的方案——框架会换代，**接口比框架长寿**。
