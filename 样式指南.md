# 项目 Markdown 样式指南

> [!NOTE]
> 本指南是本书所有文件的排版基准。新增或编辑文件时请遵循。基于 2025-2026 跨渲染器（GitHub / Obsidian / Typora / VS Code）实测的通用语法。

---

## 1. 每个文件顶部：YAML Frontmatter

```markdown
---
title: 文件标题
updated: YYYY-MM-DD
tags: [分区, 子分类]
---
```

- **必填**：`title`、`updated`
- **可选**：`tags`、`aliases`（Obsidian）、`status: stable|draft`
- **长度**：不超过 10 行
- **`updated` 字段语义**：表示**该文件内容**的最后修订日期（不是版本号 bump、不是 frontmatter 微调、不是被引用计数变化的日期）。简单规则：**只有当文件正文实际变化时才 bump `updated`**。一次 `sed` 全局替换不应该改动所有文件的 `updated`——发布流程脚本会在 commit 阶段按 `git diff` 自动判断哪些文件需要 bump。

---

## 2. 一级标题：只有一个 H1

```markdown
# 文件主标题
```

- 每个文件只出现一次 `#`
- 之后全部用 `##`、`###`、`####`
- **不要超过 h4**（超过 = 该章节应拆文件）

---

## 3. 告警 / Callouts：用 GitHub Alerts 语法

五种标准类型（GitHub / Obsidian / Typora 通用）：

```markdown
> [!NOTE]
> 一般性说明、补充信息

> [!TIP]
> 建议、最佳实践

> [!IMPORTANT]
> 重要但不危险，需读者留意

> [!WARNING]
> 需要警惕、可能出错

> [!CAUTION]
> 危险操作、会造成损失
```

**替换规则**：旧文档里所有裸 `> 🔑` / `> ⚠️` / `> 🚨` 引用块都应改为上述五种之一。

> [!IMPORTANT]
> Alerts 不能嵌套。如果需要嵌套引用，用普通 `>` 引用即可。

---

## 4. 行内 Emoji：节制使用

- **允许**：领域标记（🇨🇳 🇺🇸 🇯🇵）、产品图标（📦 📘 📖）
- **不建议**：状态标记（⚠️ ✅ ❌ 🔑）——让 callouts 承担视觉重量
- **每个标题最多 1 个 emoji**（中文 + emoji 很挤）
- **不要同时用 `[!WARNING]` 和 ⚠️**——语义重复

---

## 5. 代码块：永远带语言标签

✅ 正确：

````markdown
```bash
brew install claude-code
```

```python
import anthropic
```

```json
{"model": "claude-opus-4-7"}
```

```markdown
## 示例
```
````

❌ 错误：

````markdown
```
裸代码块
```
````

常用语言标签：`bash` · `sh` · `python` · `javascript` · `typescript` · `json` · `yaml` · `toml` · `markdown` · `sql` · `diff` · `mermaid`（GitHub 原生渲染）

---

## 6. 表格：控制宽度

**触发拆分的信号**：
- 列数 > 6
- 单元格内容 > 60 字
- 在手机 / 窄屏上横向溢出

**三种拆分方案**：

1. **按类别拆多张表** + H3 分隔
2. **用 `<br>` 软换行** 让长单元格不把表格撑宽：

    ```markdown
    | 模型 | 定价 |
    |---|---|
    | Claude Opus 4.8 | $5 in / $25 out<br>per MTok |
    ```

3. **用 `<details>`** 折叠不常看的细节：

    ```markdown
    | 字段 | 说明 |
    |---|---|
    | 详情 | <details><summary>展开</summary>长段详情文字...</details> |
    ```

---

## 7. 链接与导航

### 文件间跳转
始终用**相对路径**：

```markdown
✅ [第 6 章](../知识/06-AI自治与上下文架构约束.md)
❌ [[第 6 章]]   （Obsidian-only，GitHub 不识别）
```

### 统一的导航页脚

每个文件**底部**用以下格式（分节点不同按需增减）：

```markdown
---

← [上一章](./prev.md)  ·  [📖 目录](../README.md)  ·  [下一章 →](./next.md)
```

**根目录 README.md** 不需要 prev/next，只需要入口说明。

---

## 8. 分节符：用 `---` 而不是空行

主要章节之间用横线分隔：

```markdown
## 第一节

内容...

---

## 第二节

内容...
```

- **每个 `##` 之前**建议有 `---`
- `###` 之间不需要
- 避免连续空行（3 个以上空行）

---

## 9. 强调：克制使用粗体

- **粗体**：用于**关键结论**、**术语首次出现**
- *斜体*：用于**强调重音**，不用于大段文字
- **不要粗体整段**——噪音高于信号

---

## 10. 列表：规则

- **无序用 `-`**，不用 `*` 或 `+`（跨渲染器最稳）
- **有序用 `1.`**，不用 `1)`
- **嵌套最多 3 层**，超过请改结构
- **列表项之间可用空行**，提高可读性（牺牲紧凑度）

---

## 11. 脚注引用（适用于有大量来源的文档）

```markdown
结论 X。[^src1]

[^src1]: 作者 · 《标题》 · 发布方 · YYYY-MM-DD · URL
```

脚注统一放文件**末尾**，按出现顺序编号。

---

## 12. 文件命名

- **中文文件名** OK（macOS / Linux 无压力）
- **不使用空格**，用 `-` 分隔
- **章节文件带数字前缀**：`04-系统架构.md`、`Week1-API.md`
- **总览文件**固定名 `总览.md`

---

## 13. 最后更新标注

文件顶部 frontmatter 里 `updated: YYYY-MM-DD` 是权威来源。**不要**在正文里再写一次"最后更新于 xxx"——单一事实源原则。

---

## 14. 快照数字：单一事实源 + 就地标日期

会随时间漂移的数字（价格、模型名、阈值、榜单、容量参数）只在**定义它的那一章**写一次，并就地标快照日期；其它章节**引用**，不复制数值。

- **定义源就地标日期**：在数字出现的章节用快照标记。范本见 [深入 03](深入/03-模型与工具场景化最佳实践.md) 与 [深入 18](深入/18-LLM成本工程.md)——`> [!WARNING]` 块里写"内容快照日期为 YYYY-MM-DD……本书给出的是量级，不是精确账单"，或在小节标题加 `（YYYY-MM 快照 🕒）`。
- **派生处只引用，不钉值**：例如全书的 S/M/L 美元档以 [第 11 章标定表](架构/01-AI系统参考架构.md) 为唯一来源，第 12 / 13 / 16 章写"第 11 章 M 档区间"而不是再写一遍 `$50k–$5M`。
- **改派生必回改源**：如果你在复习卡、术语表等派生处更新了一个快照数字，**必须回头改定义它的源章节**——否则就会出现"派生物更新了、正典烂在原地"的自相矛盾。
- **结构与数值分离**：判据的**结构**（看增速、是否出过业务级事故、控制面是否稳定）写正文、永不过时；**数值**进带快照标记的表，随单价漂移。

---

## 15. 强建议带"自失效触发器"

凡是给出**强且可能被能力进步推翻**的建议（默认推荐、阈值、分级节奏、选型起点），在建议块末尾附一个 `> [!WARNING] 何时这条不再适用` 小节。

- **写成可观测的事件，不是时间**：触发条件是"当 X 发生时"（开源在你的 eval 上追平前沿、自建 TCO 结构性反转、单步可靠性跨过阈值、推理范式离开自回归 Transformer……），不是"过 N 个月后"。
- **指明该重审什么**：是微调阈值，还是整章重审。范本见 [第 13 章末](架构/03-架构师的决策框架.md)、[第 5 章 SLI 节](知识/05-AI推理服务的可靠性工程.md)。
- **理由**：见 [第 13 章规则 4](架构/03-架构师的决策框架.md)——"没有反向条件的决策不是决策，是宗教"。这条规则也适用于本书自己。

---

## 16. 术语与 part 标记

- **术语分两层**：第一性术语（致命三角、Blast Radius、Reversibility、Step Budget 这类跨工具长寿的概念）与当前产品/参数名词（模型名、档位、默认参数）分开放，后者带快照标记、标注"查用即可"。范本见 [附录 C · 术语表](附录/C-术语表.md)。
- **正文引号一律用直引号 `"…"`**：不用弯引号 `“…”`／`”…”`（历史上曾有智能引号转换只跑了部分文件，造成门面与正文风格分裂）。检查命令：`grep -n '[“”]' <file>`。
- **`part-N` tag 即"第几部分"**：frontmatter `tags:` 里的 `part-1`…`part-4` 必须与 [README](README.md) / `mkdocs.yml` nav 的部分归属一致。Unit 属第四部分 → `part-4`。调整章节归属时，记得同步这个 tag。

---

## 17. 排版检查清单（发文前过一遍）

- [ ] 有 frontmatter（title + updated）
- [ ] 只有一个 H1
- [ ] 告警块用 `> [!TYPE]` 语法
- [ ] 代码块都有语言标签
- [ ] 表格列数 ≤ 6，必要时用 `<br>` 或拆分
- [ ] 链接用相对路径，不用 wiki-link `[[...]]`
- [ ] 底部有统一导航页脚
- [ ] 主 `##` 之间有 `---` 分隔
- [ ] 没有状态 emoji 与 callout 语义重复（`⚠️` + `[!WARNING]`）
- [ ] 正文没有弯引号（`grep -n '[“”]'` 归零，§16）
- [ ] 粗体不超过正文的 ~5%
- [ ] 快照数字在定义源就地标了日期；派生处是引用而非复制（§14）
- [ ] 改了某个快照数字，已回改定义它的源章节（§14）
- [ ] 新加的强建议带了"何时不再适用"事件触发器（§15）
