# 引章 · 速览：大模型是如何工作的（使用者视角）

> [← 返回目录](README.md)

这一章只负责一件事：让你看懂后面的内容。**原理极简，够用就停**。已有底子的可以跳过。

## 一句话本质

当前的大语言模型（LLM）是一个**下一个 token 的概率预测器**。给它一段 token 序列，它返回对"下一个 token"的概率分布。反复采样，就变成你看到的"对话"。

它不是魔法，不是思考，也不是数据库——它是在**海量文本上压缩出来的统计引擎**。

理解这一点，它的能力和局限就都说得通了。

## 9 个你必须知道的机制

**1. Token**
模型不看字符，看 token。中文通常每字 1-2 个 token（中文友好 tokenizer 可低于 1 token/字），英文约 0.75 词/token。上下文长度、计费、延迟**全部**按 token 算。

**2. Context window**
一次对话能塞多少 token。当前主流 128k-1M。**不是越长越好**——埋在长上下文中段的信息最容易被忽略（"迷失在中间"现象：首尾好、中间差，机制见[科学 02](科学/02-Lost-in-the-Middle.md)）。

**3. Temperature / top-p**
控制采样随机性。Temperature=0 最确定，≥1 发散。SRE 场景多用 0-0.3，创意场景 0.7+。

**4. System prompt**
放角色、规则、约束。贯穿整个对话。**最重要的指令放最前或最后**，中间容易被稀释。

**5. Tool use（function calling）**
LLM 返回结构化请求（"调用 X 函数、参数 Y"），你的代码执行工具，把结果塞回对话，LLM 继续。这就是"让 AI 动手操作外部环境"的全部机制——**它只会描述想做什么，不会自己执行**。这一点是第 6 章致命三角的前提。Tool use、function calling、MCP 都只是协议层的意图表达；真正把意图变成系统动作的是宿主侧的 Harness。

**6. Embedding**
把文本映射为高维向量，语义相近的向量距离近。是 RAG 检索的基础。

**7. RAG（检索增强生成）**
用户问题 → embedding → 在向量库找最相似片段 → 把片段塞进 prompt → LLM 回答。解决"模型不知道你私有数据"。

**8. 流式输出（streaming）**
Token 一个个吐。改善体感延迟，但**错误处理变复杂**（输出到一半中断怎么办）。

**9. 推理 / Thinking 模式**
让模型出答案前先"想"一段（test-time compute / extended thinking）。复杂推理、架构决策上质量明显更好；代价是这段思考按**输出 token 计费**、延迟也涨——预算按 3-10× 估。延迟敏感场景别开，质量敏感场景该开。2026 年的一线旗舰基本都带它（机制与选型见[深入 03](深入/03-模型与工具场景化最佳实践.md)）。

## 能力与局限（实用版）

**擅长**
- 有上下文的语言任务：改写、总结、翻译、解释
- 结构化抽取：非结构化文本 → JSON
- 常见场景的代码生成
- 作为"判官"给其他输出打分（judge 模型）

**不擅长（一定会踩的坑）**
- 无工具辅助的严格算术与大量元素计数（推理模式能缓解，不能根治）
- 精确事实（训练截止日之后 + 长尾知识 = 会编）
- 长链条推理：错误会沿步数累积，链越长越脆——推理模式把能可靠完成的步数推远了很多，但没根治；阈值随任务而变（机制见第 4 章）
- 可靠地承认"我不知道"——有时会说，但不能把它当成设计假设

## 四种你一定会遇到的失败模式

前言那四种故障模式（静默降级、自动化放大、组织接缝、外部契约）是事故级视角；下面这四种是机制级视角——前者在系统层面爆发，往往由后者在模型层面点火。

| 失败 | 表现 | 对抗方式 |
|---|---|---|
| 幻觉 Hallucination | 流畅地编造不存在的 API、函数、事实 | RAG ＋ 强制引用 ＋ 结构化输出 |
| 截断 Truncation | 输出被 max_tokens 或上下文边界切断 | 留安全边际 ＋ 支持续写 |
| 注入 Prompt Injection | 用户/外部数据里藏了指令，被当命令执行 | 基础设施层切断外泄路径（第 6 章） |
| 漂移 Drift | 模型版本升级后同一 prompt 行为变了 | 固定模型版本 ＋ 持续 eval（第 8 章） |

## 你应该选什么服务

| 形态 | 何时选 |
|---|---|
| API（Anthropic / OpenAI / Google） | 大多数场景。免运维、能力最前沿 |
| 托管服务（Bedrock / Vertex） | 企业合规、VPC 接入 |
| 统一网关 / 模型服务层 | 多上游接入、统一鉴权 / 限流 / 计费 / 路由，并为观测与成本归因打底 |
| 本地推理（vLLM + 开源模型） | 数据敏感、追求可控、或量大到自建 TCO 反转（判据见[第 13 章决策 1](架构/03-架构师的决策框架.md)） |

作为 SRE 架构师，**先把 API 用熟，再按需下沉到本地**。不要一上来就自建——你先得知道标准模型能做什么，才知道自建要对齐到哪个基线。

> [!WARNING]
> **何时"先 API"不再是默认起点**：这条建议成立，前提是闭源 API 在能力上明显领先、且把模型送出域对你不是硬约束。两个事件会推翻它：① **开源模型在你的核心场景追平 API 前沿**（不是榜单接近，是你自己的 eval 上追平）；② **数据不出域成为硬合规约束**（金融、医疗、政府）。任一成立，合理起点就直接是"自建基线"，"先 API"反而变成绕路。判断依据是这两个事件，不是模型发布的新闻。

## B1 / B2 / B3 是什么（先备一份速查）

后面 Unit 0-5 的每周任务里会反复看到 "B1 预测 / B2 写作 / B3 阅读"——它们是贯穿全书的三个学习能力训练动作：

- **B1 预测-对比**（每日 5 分钟）：问 AI 前先猜答案，看完对比
- **B2 AI-挑错式写作**（每周 30-60 分钟）：自己写初稿，让 AI 当 critic 挑错
- **B3 无 AI 阅读块**（每周 45 分钟）：关掉 AI 读一手材料

详细做法见 [第 18 章 · 三个核心训练动作](练习/10-三个核心训练动作.md)。Unit 任务里出现这些缩写时，就是指它们。
