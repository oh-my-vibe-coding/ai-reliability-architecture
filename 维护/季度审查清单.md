# 季度审查清单

> [← 维护系统](README.md)

> [!NOTE]
> **每季度第一个月的第二周，花 2 小时**。
>
> 月度更新是 patch，季度审查是 **minor**——结构性变化、深度复查、累积改动集中消化。

---

## 预期时间：2 小时

```
Phase 1 · 扫描（30 min）: 过完整漂移度表 📊 和 🔩 章节
Phase 2 · 深度更新（60 min）: 处理积累的结构变化
Phase 3 · 整合发布（30 min）: CHANGELOG + 版本 bump + review
```

---

## Phase 1 · 扫描（30 min）

### 对照 [漂移度表](漂移度表.md)

#### 📊 快变章节（每个 3-5 分钟）
- [ ] 深入 03 · 模型与工具 — 本季度有无**新厂商**？有无**范式变化**（如 thinking 模式普及）？
- [ ] 深入 08 · 记忆 — 前沿研究方向有重大进展吗？
- [ ] 深入 10 · 事故模式库 — 累积了多少新 pattern 待加？
- [ ] 共同语言 03 · Research Eval — benchmark 生态有结构变化吗？（如某个 benchmark 饱和退役）
- [ ] 附录 B · 参考文献 — 有没有**新必读**论文？

#### 🔩 缓变章节（每个 1-2 分钟快扫）
- [ ] 知识 4-9：有没有新 SLI 要加？新架构约束出现？
- [ ] 深入 01/02/05/06/07：数字 / 工具名是否过时？
- [ ] 科学 03/04：方法有没有迭代？
- [ ] 共同语言 01/02/04/05：主流做法有没有变？

**发现待改点** → 列在临时清单。

---

## Phase 2 · 深度更新（60 min）

按 Phase 1 的发现清单，**系统性**做结构变化。和月度更新的区别：

| 维度 | 月度 | 季度 |
|---|---|---|
| **数字** | ✓ | ✓ |
| **新厂商 / 新工具** 加进表格 | ✓ | ✓ |
| **章节段落重写** | ✗ | ✓ |
| **整章删减** | ✗ | ✓ |
| **新增子章节** | ✗ | ✓ |
| **跨章关联更新** | ✗ | ✓ |

### 常见季度级变更

**Case 1 · 某 benchmark 被揭露严重污染**
- 共同语言 03 · 加警示
- 深入 03 · 相关 ranking 加注释
- 跨章节引用更新

**Case 2 · 某推理引擎出了新主版本（如 vLLM v1 → v2）**
- 深入 01 / 02 / 05 · feature 列表更新
- 代码 / 示例 · 适配（如 API 变了）
- 深入 03 · 工具生态表

**Case 3 · 一个新"必学"概念（如 2025 年的 Test-time Compute）**
- 可能单独加一个深入或科学章节
- 这是 **minor** 版本的核心场景

### 更新原则

- **保持本书气质**（精简、SRE 视角、不复制 ML 教材）
- **更新不扩张**：删一个过时的再加一个新的
- **跨章一致性**：修改一处，检查被引用的所有地方

---

## Phase 3 · 整合发布（30 min）

### 3.1 更新 CHANGELOG

Minor 版本的 CHANGELOG 比 patch 详细：

```markdown
## v1.1.0 — YYYY-MM-DD

### 新增
- [列新章节、新子节]

### 重大更新
- [描述结构性变化]

### 修订
- [小改动合集]

### 已知问题 / 遗留
- [说明什么还没改完]

### 本季度关键趋势（元）
- [1-3 句话总结本季度 AI 领域的核心变化]
```

### 3.2 Bump 到 minor

- README.md: v1.0.X → v1.1.0
- CHANGELOG.md: 对应
- 所有被触碰章节的 frontmatter `updated`

### 3.3 自检

- [ ] 漂移度表有没有过时？
- [ ] 新章节加了 frontmatter 和 nav footer 吗？
- [ ] 复习/ 的 SR 卡要不要同步补？
- [ ] Bloom 标签对新章节打了吗？

### 3.4 Review（团队维护时）

- 第二个维护者 review
- 重大结构变更要有 consensus
- 过 [样式指南](../样式指南.md) 的排版清单

## 专项：季度的"Unrelease → Release"合并

[CHANGELOG · Unreleased](CHANGELOG.md#未发布unreleased) 累积了若干月度期间"标记但未做"的待办：

- [ ] 过一遍 Unreleased
- [ ] 能在本季度做的合并
- [ ] 做不完的带到下季度

---

## 元：季度审查本身的审查

每次做完问自己：

1. 这次审查**找到的改动够不够**？如果只改了几个数字，可能没深扫。
2. 这次审查**花的时间够不够**？少于 1.5 小时可能流于表面。
3. **读者价值感**提升了多少？（如果章节变多但质量没提，是肿胀不是成长）

---

## 年度重审的种子

季度审查中如果发现**三个或更多"这块要大改"**，就记下来——**年度重审**时集中处理：

- 创建 `维护/下次年审/` 目录
- 每次发现大结构变化放一个 note 进去
- 年审时集中消化

---

## 给 SRE 的元启示

季度审查 = **服务的容量规划与重构窗口**。
月度是"今天的火扑了"，季度是"这个系统该不该重新设计了"。
