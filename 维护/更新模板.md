# 更新模板 · 如何规范记录一次更新

> [← 维护系统](README.md)

> [!NOTE]
> 不规范的更新 = 没人知道发生了什么。规范记录让**未来的你（或继任者）能追溯每次变更**。

---

## 1. CHANGELOG 条目模板（复制粘贴用）

### Patch 版本（月度级）

```markdown
## v1.0.X — YYYY-MM-DD

### 快照更新
- 深入 03 · LM Arena Overall #1: Claude Opus 4.7 Thinking → [新 #1]
- 深入 03 · SWE-bench Verified top 5 更新（附日期）
- 深入 03 · OpenAI 新增 GPT-5.6 到表格
- 共同语言 03 · 更新 benchmark SOTA 分数

### 勘误
- [列本月发现的小错误]

### 死链替换
- 附录 B · `X URL 失效` → `Y URL`（同一来源的新地址）

### 发现但未纳入
- [发现但决定不加进书的东西，留作参考]
```

### Minor 版本（季度级）

```markdown
## v1.1.0 — YYYY-MM-DD

### 新增章节
- 科学 05 · [新主题]（~300 行）
- 代码 08 · [新参考实现]

### 重大更新
- 深入 10 · 新增 Pattern 16、17、18（本季度生产事件累积）
- 共同语言 04 · 改写 §3-5（DPO → GRPO 的代表性变化）

### 修订
- [小改合集，每条一行]

### 删除 / 废弃
- [说明什么被删了、为什么]

### 本季度关键趋势（元）
- [1-3 句话总结本季度 AI 领域的核心变化和对书的影响]

### 已知问题 / 遗留
- [说明什么还没改完，下季度处理]
```

### Major 版本（年度级）

```markdown
## v2.0.0 — YYYY-MM-DD

### 破坏性变更 (BREAKING)
- [整章重构、目录调整]

### 新增分区
- [全新 Part / 目录]

### 框架变化
- [三轨 → 四轨之类]

### 迁移指南
- 老读者从 v1.x 到 v2.0 要注意什么
```

## 2. Frontmatter 更新模板

每次改动章节都要同步：

```yaml
---
title: [不变]
updated: 2026-MM-DD   ← 改成今天
tags: [...]
maintainer: [可选] your-name
---
```

**不要**改 `title`（会破坏书的引用链）。需要改 title 时走 minor 版本。

## 3. Commit Message 模板

### Patch
```
chore(v1.0.X): monthly update YYYY-MM

- 深入 03: LM Arena + OpenRouter 月更
- 附录 B: 2 死链替换
- 参考 CHANGELOG 完整列表
```

### Minor
```
feat(v1.1.0): quarterly release Q3 2026

Highlights:
- 新增科学 05 · ...
- 深入 10 新 pattern × 3
- 共同语言 04 改写 §3-5

See CHANGELOG for full list.
```

### Major
```
feat!(v2.0.0): annual framework revision 2026

BREAKING: 目录重构，老引用需更新
See CHANGELOG for migration guide.
```

## 4. "更新时要同步更新的其他地方" 清单

改一个章节，经常漏的关联点：

- [ ] **主 README.md** 的目录链接（改名时）
- [ ] **CHANGELOG.md** 的对应版本条目
- [ ] **漂移度表.md** （如果漂移级别变了）
- [ ] **复习/核心概念卡.md** （如果有相关事实变了）
- [ ] **复习/anki-import.csv** 同步
- [ ] **复习/Active-Recall题库.md** 可能要改题
- [ ] **其他章节的 cross-link**（grep 一下老章节名）
- [ ] **样式指南.md** 若引入新语法

一次更新 **不改这些** = 留下未来的坑。

## 5. 记录 "发现但没做" 的机制

很多时候月度扫到的东西不值当一次 patch，但也不能忘：

**创建 `维护/待办.md`**（按需）：

```markdown
# 待办池

## 待下次 patch
- [ ] 深入 04 · 某数字从 14k → 16k（待验证）

## 待下次季度
- [ ] 考虑加一节"多模态推理的 SLO"
- [ ] 科学 02 补一张 RULER 真实曲线图

## 待年度重审
- [ ] 三条主线是否需要演进成四条？
- [ ] "共同语言"板块是否需要按 ML 新角色细分？

## 放弃不做
- [ ] ~~加 LLM-native database 章节~~（定位不符，放弃）
```

**每次更新时过一眼这个文件**，消化能做的，补充新的。

## 6. "不该做"清单（避免踩坑）

- ❌ **改 title 不 bump minor**：title 是 URL 和外部引用的一部分，改了要 bump
- ❌ **CHANGELOG 只写"更新了一些章节"**：具体到章节和变化
- ❌ **frontmatter 不改 updated**：读者无法感知新旧
- ❌ **一次 commit 合并 patch + minor 变更**：分开，便于回滚
- ❌ **删章节不在 CHANGELOG 说**：读者会困惑"那一章去哪了"
- ❌ **重命名目录**（如 `深入/` 改名）：破坏所有 cross-link，极度避免

---

## 7. 自检清单（每次更新后）

- [ ] CHANGELOG 有对应条目？
- [ ] 版本号 bump 了？
- [ ] **README.md 和 维护/README.md 的版本号都改了？**（常被漏）
- [ ] 被改章节的 frontmatter updated 改了？
- [ ] 相关 cross-link 都还有效？
- [ ] 新内容走过 [样式指南](../样式指南.md)？
- [ ] **新增的价格 / 模型名 / 榜单数字都带了"快照日期 + 官方源链接"**？
- [ ] **改了某个快照数字，回改了定义它的源章节了吗**？（派生处更新、正典烂在原地是最常见的自相矛盾）
- [ ] **新加的强建议带了"何时不再适用"事件触发器吗**？（见 [样式指南 §15](../样式指南.md)）
- [ ] 如果改了事实，复习/ 里的 SR 卡同步了？
- [ ] 本地 preview 一下渲染效果？
