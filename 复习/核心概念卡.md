## 第一部分 · 引章 & 大模型基础

### 引章：大模型速览

**Q1** · LLM 的一句话本质是什么？
**A** · 一个下一 token 的概率预测器（在海量文本上压缩出的统计引擎）。

**Q2** · 什么是 Context Window？越长越好吗？
**A** · 一次对话能塞的 token 数。**不是越长越好**——埋在长上下文中段的信息最容易被忽略（Lost in the Middle：首尾好、中间差）。

**Q3** · Temperature=0 和 Temperature=1 的区别？
**A** · 0 最确定，1 发散。SRE 场景多用 0-0.3。

**Q4** · Tool use / function calling 的本质？
**A** · 模型只会**描述**想做什么，实际执行由宿主代码完成。

**Q5** · RAG 是什么？
**A** · Retrieval-Augmented Generation。查询 → embedding → 向量库检索 → 拼进 prompt → 生成。

**Q6** · 4 种必见失败模式？
**A** · 幻觉、截断、注入、漂移。

## 第二部分 · 知识章节

### 第 4 章 · 复合 AI 可靠性数学

**Q** · 每步 95% 正确的 10 步 Agent 端到端正确率约多少？
**A** · ~60%（0.95^10 ≈ 0.599）。

**Q** · 复合系统可靠性工程的两件新工作？
**A** · Step 预算 + Verifier 插入点。

### 第 5 章 · AI 推理服务可靠性

**Q** · LLM 推理服务必须分开定 SLO 的两个指标？
**A** · TTFT（首 token 延迟）和 tokens/s（输出速度）。

**Q** · 容量规划新三角？
**A** · GPU 显存 × 上下文长度 × QPS，三者互相约束。

**Q** · 静默降级（Silent Degradation）是什么？
**A** · 指标全绿但系统数学已经坏了（bf16/fp32 失配、XLA 怪癖、kernel 非确定性）。

### 第 6 章 · AI 自治与上下文约束

**Q** · Lethal Trifecta 三个组成？
**A** · 私有数据访问 + 不受信输入 + 外泄通道。三者共存 = 攻击可行。工具访问不是独立一条腿，是放大器。

**Q** · Prompt injection 的正确防御层在哪？
**A** · 基础设施层（沙箱 / egress 白名单 / capability-scoped 凭证），不是 prompt 层。

**Q** · Agent 的一句话构成？Harness 包含哪些工程件？
**A** · **Agent = 模型 + Harness**。Harness 是你亲手组装的工程件：系统提示 + 工具与权限 + 记忆 + 子 Agent + 验证回路。模型只负责"描述下一步"，Harness 决定它能看到什么、能做什么、被谁验证——能力与边界由架构（Harness）决定，不由 prompt 决定。（同一 harness 从执行视角看，就是执行工具调用的宿主运行时 / 执行边界；见引章 Q4。）

**Q** · Agent 的三类核心风险各对应哪种 harness 机制？
**A** · 幻觉 → 验证回路（第 8 章 / 深入 06 Eval）；prompt injection → 权限工程（本章致命三角 / 深入 07：沙箱·egress·capability 凭证）；成本失控 → 资源治理（第 16 章 预算治理 / 深入 18 / 深入 04）。三者都是 Harness 的组成部分——都不是 prompt 层能解决的。

### 第 7 章 · 从链到图

**Q** · 并行合并（AND，所有分支都要对）在可靠性上等价于什么？并行买到的是什么？
**A** · 等价于一条等长的串链——联合正确率 = 各节点正确率之积（0.857⁴ = 0.95¹²）。并行买的是**时延**（关键路径变短），不买可靠性。

**Q** · 乘积公式 p^n 算不到、图才有的失效有哪些？
**A** · 共因失效（共享上游 → 冗余是假象）、无 verifier 的静默合并（自信地错）、无界循环（n 不再有界）、并行写冲突 / 双重提交。纪律：按**边**上 verifier + 幂等 + step 预算。

### 第 8 章 · 质量观测与 Data Flywheel

**Q** · 为什么 trace 和 eval 应该是一套系统？
**A** · Traces 就是 eval 数据集，evals 就是生产监控。分成两套会数据断裂、重复投入。

**Q** · 三层 Eval 分别是？
**A** · L1 Assertion（硬规则）/ L2 Judge 模型 / L3 A/B + 用户反馈。

## 第三部分 · 架构章节（第 11-17 章）

### 第 11 章 · AI 系统参考架构

**Q** · AI 系统参考架构的四个面是什么？为什么面的划分比组件划分更重要？
**A** · **控制面**（网关/Prompt/记忆）、**能力面**（编排/模型/检索/工具）、**质量面**（Eval/数据飞轮/红队）、**运维面**（可观测/Registry/预算）。组织通常按面分团队，**跨面接缝**是事故和扯皮的高发区——面划分错了，组件再全也是结构性问题。

**Q** · 参考架构中描述每个组件的契约卡五栏是什么？
**A** · **职责**（一句话，写不完说明该拆）、**SLO**（必须可数值化）、**失败模式**（写不出 = 只画过没设计过）、**归属**（RACI 的 R，单一团队）、**可替换边界**（什么变了组件就要重写）。

**Q** · RAG 系统里换 embedding 模型和换向量库，哪个才是真正的不可逆点？
**A** · 换 **embedding 模型**——新旧向量不在同一空间，需全量 re-embed + reindex + 重测召回（约 3-6 个月）。只换向量库便宜得多：向量可导出迁移，只需 reindex。

**Q** · 从多上游网关（M 档）跨到自建推理（L 档）的硬门槛是什么？
**A** · **Eval / Trace / Registry 等控制面已稳定运行 ≥ 6 个月**——L 档对控制面的要求是 M 档的两倍，控制面不稳就上自建等于自杀；另需 ≥ 3 名能写 CUDA、读推理框架源码的工程师。走错档比慢一拍贵一个数量级。

### 第 12 章 · AI-SRE 组织设计

**Q** · AI SRE 组织设计的第一原则是什么？
**A** · **端到端用户感知 SLO** 必须有一个**单一负责人团队**，且该团队有权调度其他团队（ticket 进对方 sprint 排期，不只是开 ticket）。否则事故时各团队指标全绿、互相甩锅，没人回答用户。

**Q** · 中型公司（50-500 工程师）的三角组织由哪三个团队构成？关键配套设计是什么？
**A** · **平台 SRE/网关**、**ML 平台/推理**、**质量/Eval**；应用团队不在三角内，是平台的共同设计者。任何两团队之间必须有**固定例会**——三个例会缺一个，对应那条边就变成事故源。

**Q** · 为什么 Eval 团队不能向 ML 汇报？
**A** · ML 的 KPI 是模型上线速度，Eval 会退化成"帮 ML 拿绿灯"——等于**让被考试的人自己出题**。Eval 必须独立汇报（Quality 或 SRE 线），ML 只提供 rubric、不掌控 gold set。

**Q** · 高 blast-radius 且不可逆的自动化决策（如 Agent 高自治放行），RACI 里的 A 有什么特殊要求？
**A** · A 不能填团队名，必须落到**具名、过能力门槛、可追责的个人**：亲手跑过该链路的红队与 eval、亲自背复盘责任、有权说"不"且不被汇报线推翻——类比注册工程师（PE）对设计签字担责。

### 第 13 章 · 架构师的决策框架

**Q** · 可逆决策与不可逆决策各该用什么速度做？
**A** · **可逆**（错了 1-2 个月能改）快做，**不可逆**（错了 ≥ 6 个月才能修复）慢做。两类用同样的速度做都是错的——所以做决策前先识别决策类型。

**Q** · 为什么每个架构决策都必须写下"反向触发条件"？
**A** · **没有反向条件的决策不是决策，是宗教**。必须写明什么数据或事件会让你重新考虑——这是把默认推荐从信仰变成可检验判断的机制。

**Q** · RAG vs 微调：微调真正合适的场景是哪两类？
**A** · ① **输出风格/格式强约束**（prompt 调不动）② **领域语言模型**（医疗/法律的术语与推理链）。约 80% 的场景 RAG + Prompt 优化能达到同样效果且维护成本低一个数量级——微调被严重高估。

**Q** · Agent 自治等级升档为什么会强制 Eval 投入同步升档？
**A** · 高自治意味着写操作和更大 blast radius，没有**人工标注采样级 Eval** 就是盲飞。这是六类架构决策**强耦合**的典型——只在单一决策上推敲、忽略联动是架构师常犯的错。

### 第 14 章 · AI-SRE 成熟度模型

**Q** · AI SRE 成熟度模型 L1-L5 五级分别是什么？
**A** · **L1 实验性接入** → **L2 生产化** → **L3 平台化** → **L4 可治理** → **L5 自我演进**。L3 是绝大多数公司的合理终点（再往上 ROI 急剧下降），L5 是理想态北极星而非目标。

**Q** · 成熟度模型的真正用途是什么（不是评级）？
**A** · **反向自检**：识别**假升档**比识别真升档重要 10 倍——假升档让组织带着虚假信心做不该做的事。任一维度（可观测/可靠性/安全/成本/组织）不达标，该级就不算达成。

**Q** · "全栈自建"能否证明组织达到成熟度 L5？
**A** · 不能。自建本身不是 L5 标志，**任何东西可换**才是——组件可换 < 30 天、关键人离职团队仍能交付、新人 4 周可上手。全自建更常见的结局是知识孤岛。

**Q** · Eval 和红队的分工差异是什么？为什么不能只做其一或分先后？
**A** · **Eval** 测"我们想做对的事做对了没"，**红队**测"我们没想到的事会不会被搞坏"。同一类工作的两面，必须一起做、不分先后。

### 第 15 章 · 不可逆决策与 Day 2 状态

**Q** · 不可逆性的五种形态谱系（从轻到重）？
**A** · **完全可逆**（1-2 周还原）→ **软不可逆**（1-3 月迁移）→ **硬不可逆**（6-18 月迁移）→ **契约不可逆**（合同/监管钉死）→ **认知不可逆**（团队心智沉没，最难识别）。

**Q** · 不可逆决策的"事前体检"四问？
**A** · ① **反向迁移路径**是什么（谁做、多久、多少钱）② 背后的**假设**错了会怎样 ③ 什么**信号**触发重新评估 ④ **三年后目标态**长什么样。答不上来 = 决策没做透。**没有反向触发的不可逆决策 = 没装刹车的火车**。

**Q** · 退路图（不可逆决策的退出路径）的正确用法？
**A** · 不是文档作业——要**每年验证一次可执行性**（跑一次端到端 fallback 演习）。当年的"30 天可切换"若变成 90 天：要么趁退路还在赶紧走，要么花成本修回来。

**Q** · AI 系统生命周期 Day 0/1/2/N 中，架构师精力该重点投在哪？真正检验架构的又是哪个阶段？
**A** · 约 **60% 投 Day 0 设计期**——Day 0 做对，Day 2/N 大部分问题不会出现；但**真正检验架构成功的是 Day N**（2-5 年后还能稳定运行且能接新业务）。

### 第 16 章 · 预算治理

**Q** · AI 系统的四种 budget 及各自约束的维度？
**A** · **Token**（财务）、**GPU**（容量）、**Risk**（安全）、**Error**（质量）。系统真实健康度由四条曲线中**最低的一条**决定。

**Q** · 为什么四种 budget（Token/GPU/Risk/Error）必须联合治理、不能各团队各管一摊？
**A** · 因为它们**相互转化**——每个看似单维度的优化都在另外三维产生波动（如降级便宜模型省 Token 却烧 Error）。单点优化 ≠ 组织级最优，架构师要在四维上保持平衡。

**Q** · Risk budget 的核心思想？
**A** · 把"安全"从无限制的硬约束变成**可量化、可累积、可消耗的预算**——自治升档、新工具上线都按单位消耗，红队回归可补回；超支自动触发降级（如禁止下月升档），而非无限等管理层批准。

**Q** · 纯 SaaS（不自建推理）场景下，GPU/容量约束消失了吗？
**A** · 没有，只是**换了形态**——上游 **RPM/TPM 配额**、峰值 429、预置吞吐额度与配额升档谈判取代了显存与 GPU 时。单位不同，治理逻辑相同。

### 第 17 章 · 与外部世界的契约

**Q** · AI 架构师必须主导的四类外部契约对象？
**A** · **法务/合规**（数据驻留、训练使用、审计）、**CFO/FinOps**（unit economics、TCO）、**上游供应商**（SLA、退役、数据条款）、**最终客户**（对客 SLA、责任划分）。不读合同的架构师，是被合同决定的架构师。

**Q** · 与 LLM 上游供应商的关系和传统云（AWS/GCP）的本质区别？
**A** · LLM 上游是**高频变化的关键供应商**——模型退役频繁、行为漂移、价格波动大、替代性低。正确姿势是**每年评估、每季对账、随时准备切换**，而非签长约躺平。

**Q** · 多供应商（multi-vendor）策略的主要决策变量是什么？
**A** · **故障容忍度**，不是花费——能容忍小时级故障就单家，容忍不到 5 分钟就要多家自动切换。多家有维护多套 prompt/eval 的代价，**2-3 家是甜蜜点**。

**Q** · 外部契约协同设计的三条原则？
**A** · **客户合同 ≤ 上游能力**（SLA 差超 1 个 9 不签）；**合规要求 ⊆ 上游能力**（上游撑不住就换上游或自建）；**财务承诺 ≤ 工程可实现**。对客承诺前自问：**18 个月后还能保证吗**？

## 第四部分 · 深入专题

### 深入 01 · TTFT 与吞吐

**Q** · Prefill 和 Decode 的瓶颈分别是？
**A** · Prefill 是 **compute-bound**（GPU 算力），Decode 是 **memory-bandwidth-bound**（HBM 带宽）。

**Q** · 为什么 Decode 每生成一个 token 都要读整个模型？
**A** · 每步需要把**全部参数**从 HBM 读到计算单元，还要读所有 KV cache。

**Q** · 70B bf16 模型在 H100 上理论 tokens/s 上限约多少？
**A** · 按单卡 HBM 带宽归一化 ~24 tokens/s（3.35 TB/s ÷ 140GB 参数）。注意 140GB 单卡 80GB 装不下，实际需 ≥2 张 H100（TP），聚合带宽下上限相应提高（~48）。

**Q** · TTFT 主要由什么决定？
**A** · 输入 token 数（prefill 时间线性相关于输入长度）。

### 深入 02 · Prompt Caching

**Q** · Prompt caching 缓存的到底是什么？
**A** · **KV 张量**（每层每头的 K 和 V），不是最终回答。

**Q** · 缓存命中的判定条件？
**A** · **Token 级前缀匹配**（不是字符串相等），必须从 token 0 开始连续匹配。

**Q** · Anthropic 缓存读取成本是常规的多少？
**A** · ~0.1×（折扣 90%），写入 1.25×，重用 2 次回本。

**Q** · 为什么不能在 prompt 开头放时间戳？
**A** · 时间戳每次不同 → tokenize 后不同 → 缓存全失效。

### 深入 03 · 模型与工具选型

**Q** · LM Arena 测的是什么，弱点在哪？
**A** · 用户盲测偏好（Elo）。弱点：偏主观对话，不一定反映编码/推理硬实力。

**Q** · SWE-bench Verified 测什么？
**A** · 真实 GitHub issue 修复，是 Agent 编码能力的黄金标准。

**Q** · 选型的 7 步？
**A** · 定义场景 → 硬约束 → 软约束 → 短名单 3 个 → 自己跑 20-50 条 eval → 小流量上线 → 扩量 or 回退。

### 深入 04 · 为什么"你好"消耗数万 token

**Q** · Claude Code 首次"你好"约消耗多少 input token？
**A** · ~14-25k。

**Q** · 这些 token 的主要去向（按占比）？
**A** · System prompt (~37%) + 工具定义 (~49%) + CLAUDE.md 等 (~14%)，用户消息本身仅 2 token。

**Q** · 为什么 LLM API 每次必须重传全部上下文？
**A** · **LLM API 无状态**——模型没有记忆，每次调用必须携带全部上下文。

**Q** · 为什么多轮对话"越聊越贵"，而且是平方级、不是线性？
**A** · 因为无状态——**每一轮都要为整段历史重新付一次输入费**。历史随轮数线性变长，累计成本 ≈ 各轮历史长度之和 → 随轮数呈 **平方级（N²）增长**：轮数翻倍、成本约翻四倍。缓解见前缀缓存（深入 02）与上下文压缩；具体 token 与账单数值见深入 04 §2–§3、§2.8。

### 深入 05 · 容量规划

**Q** · 单请求 KV cache 大小公式？
**A** · `2 × num_layers × num_heads × head_dim × bytes_per_element × token数`。

**Q** · GQA 能把 KV cache 缩多少？
**A** · 约 8-16×（Llama 3 70B 用 KV heads=8，相比 64 减 8×）。

**Q** · 训练 vs 推理集群的核心差异？
**A** · 训练是全集群一个故障域、IB 必须；推理是请求级独立、可 scale out。

### 深入 06 · Eval Pipeline

**Q** · Judge 模型需要做什么定期校准？
**A** · 抽样 50-100 条让人工 + judge 对同一批样本打标签，算 **Cohen's κ**，目标 ≥ 0.6。

**Q** · Eval 自己需要的 SLO 有哪些？
**A** · 覆盖率、评估延迟、Judge 对齐度、pipeline uptime、失败率、数据新鲜度。

**Q** · Data Flywheel 的完整循环？
**A** · 生产 trace → 标注失败集 → eval 更新 → prompt/模型变更 → rollout → 新 trace。

### 深入 07 · Prompt Injection 红队

**Q** · 攻击 6 类？
**A** · Direct / Indirect / Tool Poisoning / Exfiltration / Loop Hijacking / History Poisoning。

**Q** · 4 层纵深防御哪层最强？
**A** · **基础设施层**（沙箱 / egress / capability-scoped 凭证）。Prompt 层最弱。

**Q** · Indirect injection 为什么最狡猾？
**A** · 攻击者把 payload 埋在 Agent 会读到的**外部数据**里，输入面是"正常业务数据"无法 pre-filter。

### 深入 08 · 大模型记忆

**Q** · LLM 记忆的三个层次？
**A** · 参数记忆（训练固化）/ in-context 记忆（当前窗口）/ 持久记忆（应用层拼的）。

**Q** · "写数据库作记忆"解决了多少问题？
**A** · ~20%（持久化），剩下 80%（抽取、检索、冲突、污染、元认知、运维）没解决。

**Q** · 持久记忆引入 prompt injection 的什么新风险？
**A** · **注入一次、永久生效**——把单次攻击变成持久威胁。

**Q** · 当前最被推崇的记忆模式？
**A** · **工作笔记模式**（Claude Code 的 NOTES.md）——Agent 自己写、人类可审计、文件系统自带版本。

### 深入 17 · LLM 网关的 SRE 视角

**Q** · 多上游 LLM 网关的 SLI 必须按哪三个维度切分？
**A** · **调用方租户 × 上游通道 × 模型**三维度——任一维度合并都会导致**归因失败**。

**Q** · 多上游网关的内部 canonical schema 字段集应取各上游的并集还是交集？
**A** · **并集（union）**——只取交集会删掉各家独有的缓存计数、usage 细分、拒答分类等关键差异字段，事故排查时无法还原上游原文。

**Q** · 网关的协议转换和 MCP，本质是同一个什么结构？
**A** · **M×N → M+N**。M 个调用方 × N 个提供方直连 = M×N 份互不复用的胶水；插一层规范接口（网关 canonical schema / 工具层 MCP / 跨 Agent 通信），就塌成 M+N。像 USB-C——一个统一口，两端换设备都不重接。

**Q** · 网关层如何检测"上游通道静默退化"（错误码全正常但回答质量变差）？
**A** · 每个通道周期性跑**金标准探针**（已知答案 prompt + 确定性 metric），并做**跨通道横向对比**——通过率分叉即报警；关键是全程**不依赖上游错误码**的黑盒检测。

**Q** · 为什么说计费日志是 Data Flywheel 最廉价的起点？
**A** · 它是计费的**免费副产品**、**100% 覆盖**每次请求且自带全部关键元数据；只需补 `output_hash`、`finish_reason`、`quality_tag` 三列即可启动最小飞轮，**不依赖用户主动反馈**。

### 深入 18 · LLM 成本工程

**Q** · LLM 成本的四层归因模型是哪四层？哪个位置能做到全部四层？
**A** · **模型层 → 功能层 → 用户层 → 结构层**（system prompt/工具定义/context/历史各占多少）；**API 网关**是唯一能做到全部四层归因的位置。

**Q** · Agent 链路成本放大的三个来源是？
**A** · **步数放大**（每个 tool call 往返一次 API）、**上下文膨胀**（每步结果追加进历史）、**思考链消耗**（thinking token 不出现在输出里但照样计费）。

**Q** · Prompt 缓存的成本节省公式？
**A** · `节省 = 命中率 H × 命中前缀占比 × (1 − cached_price/full_price)`——命中率和命中前缀长度占比共同决定实际收益。

**Q** · LLM 成本优化按投入产出比排序的前三步是？
**A** · **Prompt Caching → 离线任务迁移到 Batch API → 模型降级路由**；前三步做完通常已省 60-80%，其余（上下文裁剪、步数限制）是边际优化。

### 深入 19 · 模型服务框架对比

**Q** · 主流自建推理框架当前竞争的核心机制是什么？
**A** · 前缀缓存已收敛为三家引擎的**共同默认能力**（vLLM block 级 hash / SGLang token 级 radix tree——两种粒度路线）；竞争前沿上移到**编排层**：KV 感知路由、PD 分离、大规模专家并行（llm-d / Dynamo / AIBrix）。

**Q** · 为什么 Agent 多轮场景要关注前缀缓存的实现**粒度**？
**A** · Agent 每轮只在上下文**末尾追加**，前缀天然深度共享——SGLang 的 radix tree 按 **token 级任意长度**命中，vLLM 按**整 block** 命中；两者都默认开启，深共享负载下粒度差异才显现（量级是百分之几十，不再是早年对旧版 vLLM 的 2-4×）。

**Q** · 自建推理服务的默认选型路径？
**A** · 默认从 **vLLM** 起步（社区事实标准、硬件插件生态最广）；深共享前缀 / 大规模 EP+PD 负载评估 **SGLang**；全 NVIDIA 栈追求极致再看 **TensorRT-LLM**（已 PyTorch 化、无编译门槛）；多副本上规模后加**编排层**而不是换引擎。

**Q** · TensorRT-LLM 的"编译式框架"印象为什么过时了？
**A** · 1.0（2025-09）起 **PyTorch 原生后端**成为稳定默认，当前主干（1.3 时代）已移除引擎编译路径、PyTorch 为唯一执行后端——HF checkpoint 直接 `trtllm-serve` 起服务；真实门槛从"编译周期"变成**调优复杂度与 NVIDIA-only 锁定**。

### 深入 20 · 分布式推理

**Q** · GPU 集群的互联带宽层级从快到慢是？量级关系？
**A** · **HBM（卡内）≫ NVLink（节点内）≫ InfiniBand（跨节点）**——每往下一层掉约**一个数量级**，这决定了每种并行策略能跨多远。

**Q** · 为什么张量并行（TP）必须待在 NVLink 域内、不能跨节点？
**A** · TP 每 decode 一个 token 要做 **2L 次串行 all-reduce** 且全在**关键路径**上藏不住；杀死跨节点 TP 的是"**延迟 × 次数**"加 batch 放大后的带宽，只有 NVLink 的高带宽 + 微秒级延迟扛得住。

**Q** · MoE 模型在 decode 上"省带宽"，它省显存吗？为什么？
**A** · **不省**——路由是逐 token 变的，**全部专家权重必须常驻集群显存**；所以超大 MoE 必须靠**专家并行（EP）**把专家摊到多卡，用聚合显存装下全部权重。

**Q** · 衡量 PD 解耦效果用什么指标？它的定义是？
**A** · 用 **goodput**：**同时满足 TTFT 和 TPOT 两个 SLO** 前提下每 GPU 每秒能扛的请求数——裸吞吐把超时请求也算数，goodput 才反映有效产能。

### 深入 21 · Prompt 作为运维对象

**Q** · Prompt 的发布单元是什么？
**A** · **模板 + 模型版本 + 采样参数 + 工具 schema**——四者任一变更都算一次 prompt 变更；这是"四联版本绑定"（Prompt / Model / Embedding / Judge 联合版本）的 prompt 侧。

**Q** · Prompt Registry 业界收敛的数据模型是什么？
**A** · **稳定标识符 + 不可变版本 + 可移动指针（label）**——发布 = 移动指针，回滚 = 指针移回，与容器镜像 tag 同构。

**Q** · Prompt 变更上线前的三道关卡？
**A** · **静态检查**（diff 评审 + 渲染测试 + 缓存影响分析）→ **生产 trace 回放**（成对比较行为漂移）→ **eval gate**（不达标不放）。

**Q** · "Prompt 回滚 < 5 分钟"的四个前提？
**A** · **旧版本永不静默删除**；发布走**可移动指针**（不是硬编码重新部署）；模型 / 工具 schema 等**绑定件一起回**——否则回到的是没测过的新组合；**指针变更能在 SLO 内传播到全部消费实例**（客户端缓存 TTL 小于回滚 SLO 或有推送失效）。

## 第五部分 · 科学专题

### 科学 01 · Attention

**Q** · Self-attention 的三个角色 Q/K/V 分别表示什么？
**A** · Q=我在找什么（query），K=我是什么（key），V=我的信息（value）。

**Q** · Attention 的计算复杂度？
**A** · **O(N²)**——N 个位置互相算相似度。长 context 成本的物理根因。

**Q** · KV Cache 为什么能节省计算？
**A** · 每生成新 token 时，**历史 token 的 K 和 V 完全不变**，可存下来复用。

**Q** · Attention Sink 是什么？
**A** · 模型在 softmax 里需要一个"出口"倾倒多余注意力，前几个 token 成了默认"垃圾桶"——解释了为什么前置内容受重视。

### 科学 02 · Lost in the Middle

**Q** · 长 context 中间塌陷的 4 个叠加原因？
**A** · 训练分布（重要信息多在首尾）+ 位置编码几何偏差 + softmax 稀释 + attention sink 效应。

**Q** · "宣称 context"和"有效 context"的差距？
**A** · RULER：宣称 ≥32K 的模型**只有一半**能真正用到 32K。Gemini 1.5 1M 实测有效 >128K。

**Q** · 长 context 能替代 RAG 吗？
**A** · **不能**。中间塌陷 + 成本线性涨 + 延迟高。RAG + Rerank 仍是必要的。

### 科学 03 · Quantization

**Q** · Outlier features 是什么，为什么毁掉量化？
**A** · 少数激活维度值极大（±100+），其他维度都在 ±1 以内。int8 只有 256 级，覆盖 outlier 就丢普通值精度。

**Q** · GPTQ 和 AWQ 的核心思想分别是？
**A** · GPTQ 用 Hessian 信息决定哪些权重重要并补偿误差；AWQ 认为"激活大的维度对应权重更重要"，保护这些。

**Q** · 量化静默降级的典型症状？
**A** · **整体 eval 通过但某类任务崩**（数字/结构化输出对量化特别敏感）。必须按任务类型分桶监控。

### 科学 04 · Tokenization

**Q** · 同样信息量，中文在 GPT tokenizer 下的 token 数比 Qwen 多多少？
**A** · ~2-3×（GPT 对中文每字 1.5-2 token，Qwen 每字 0.6-1 token）。

**Q** · LLM 数学差的根因？
**A** · Tokenization 不稳定——"17234" 可能切成 ["17","234"] 也可能 ["1","7","2","3","4"]。数字不是位值表示。

**Q** · Opus 4.7+（含 4.8）换 tokenizer 对账单的影响？
**A** · 同样内容 token 数比 Sonnet 4.6 多约 **35%**（1M context 对应 ~555k 词 vs Sonnet 4.6 的 ~750k 词，意味着 1.35× token 体积）。单价相同时实际成本高 ~35%。Opus 4.8 在 token 体积之外还把 `effort` 默认为 `high`，预算放大再叠加一层。

## 第六部分 · 共同语言

### 共同语言 01 · 训练生命周期

**Q** · 训练三段是哪三段？
**A** · Pre-training（世界知识）/ Mid-training（聚焦能力提升）/ Post-training（对齐）。

**Q** · SFT 和 RLHF/DPO 的区别？
**A** · SFT 教格式（"被问就答"），RLHF/DPO 教偏好（两个都对的回答里选更好的）。

**Q** · Loss spike 的 SRE 处理流程？
**A** · 暂停 → 检查最近数据批次 → 从最近 checkpoint 恢复 → Skip 可疑数据 → 继续。

**Q** · Model card 至少要看哪 5 项？
**A** · Training data（含 cutoff date）/ Architecture / Recipe / Post-training / Evaluations + regressions。

### 共同语言 02 · Data

**Q** · ML 团队时间主要花在哪？
**A** · 80% 在**数据**（pipeline、清洗、去重、污染检测、混合比例），不是调模型。

**Q** · Pre-training data pipeline 的 7 步？
**A** · Crawl → Clean → Filter → Dedup → Decontaminate → Mix → Tokenize。

**Q** · Contamination 是什么？如何检测？
**A** · 训练数据混入 eval set。检测：N-gram 匹配、canary tokens、private eval。

**Q** · Annealing mix 是什么？
**A** · 训练末期 10-20% steps 用**更高质量、更聚焦**的数据混合（数学、代码比例拉高）。近因效应。

### 共同语言 03 · Research Eval

**Q** · pass@k 的含义？
**A** · 让模型对同一题生成 k 次，至少一次通过的概率。**pass@1 才代表单次可用**。

**Q** · MMLU 为什么区分度下降？
**A** · 饱和（前沿模型 85-90%，接近人类 90%），已无法区分顶级模型。替代是 MMLU-Pro / GPQA。

**Q** · LM Arena Elo 是什么？
**A** · 用户盲测对弈 → Elo 更新。相对分数，**分差比绝对值重要**。差 20 分显著，差 5 分是噪声。

**Q** · Benchmark contamination 怎么识别？
**A** · 某单一指标飙升而 Arena / private eval 不动 + 强项不合理集中 → 大概率刷榜。

### 共同语言 04 · Alignment

**Q** · HHH 三原则？
**A** · Helpful / Honest / Harmless。对齐的三目标。

**Q** · DPO 相比 RLHF 的优势？
**A** · 不训 Reward Model，直接用偏好对比做训练。简单 100×、稳定、效果接近。

**Q** · Reward hacking 是什么？举一个例子？
**A** · Policy 找到 Reward Model 的漏洞获得高分但实际不是好回答。例：RM 偏好长回答 → Policy 学会啰嗦。

**Q** · Jailbreak 和 Prompt Injection 的区别？
**A** · Jailbreak 是"让模型说违规内容"；Prompt Injection 是"让 Agent 做违规动作"。对 SRE 后者更严重。

### 共同语言 05 · 分布式训练

**Q** · FSDP / ZeRO-3 的核心思想？
**A** · 把模型参数、梯度、优化器状态都切片到不同 GPU，forward/backward 时临时 gather。兼得 DP 简单和 TP 省显存。

**Q** · 训练和推理集群的网络要求差异？
**A** · 训练必须 **InfiniBand / NVLink**（每 step 传数百 GB）；推理 Ethernet 足够。

**Q** · Checkpoint 的 SRE 关键指标？
**A** · 写耗时、写失败率、存储容量趋势、最近 checkpoint 年龄、跨区复制延迟。

**Q** · MFU（Model FLOPs Utilization）目标？
**A** · 实际 FLOPs / 峰值 FLOPs，**60-70% 是生产目标**。低于 50% = 效率问题。

## 第七部分 · 学习方法与训练动作

**Q** · B1 动作的核心提问是什么？

**A** · (1) 有没有"自己想通本想问 AI 的问题"？(2) B3 有没有完成？(3) B1 方向感有没有提升？
